# Architecture Methods Catalog

说明：

1. 本文档参考 `skills/_shared/brain-methods.csv` 的“方法名称 + 描述”组织方式，为 architecture 阶段提炼可直接落地的方法论目录。
2. 适用范围优先覆盖 `G202`（架构愿景）、`G201`（技术策略）、`G203`（架构蓝图）与 `G204`（架构评审验证）。
3. 这里的方法定义用于统一术语、适用阶段和最小执行提示；具体在哪个步骤使用，以各 SKILL 的 `SKILL.md` 和 `template.md` 为准。

| category | technique_name | applicable_skill | description |
|---|---|---|---|
| architecture_vision | 系统上下文图 | `G202` | 从系统边界、外部参与者、上下游系统和关键交互出发定义问题边界，确保后续架构讨论不脱离实际上下文。 |
| architecture_vision | 架构驱动识别 | `G202` | 从业务目标、质量属性、约束和关键场景中识别真正驱动架构的因素，并做优先级排序。 |
| architecture_vision | 质量属性场景 | `G202` | 用场景化方式描述性能、可靠性、安全、扩展性等质量目标，至少明确场景、刺激、响应和边界。 |
| architecture_vision | 质量属性效用树 | `G202` | 将多个质量属性按业务价值和架构影响做优先级展开，帮助识别最应该优先支撑的属性。 |
| architecture_vision | 约束分层法 | `G202` | 将约束拆分为业务、技术、合规、交付、组织等层次，区分强约束与弱约束，避免把假设误写成硬边界。 |
| architecture_vision | 风险驱动分析 | `G202` | 以主要风险和不确定性为牵引，先识别最可能影响架构方向的风险，再决定需要重点收敛的边界。 |
| architecture_vision | 场景优先级排序 | `G202` | 从核心主场景、高风险场景、异常场景中排序，避免把所有场景都当成同等重要的架构输入。 |
| architecture_vision | 假设清单与验证计划 | `G202` | 将当前尚未证实的关键判断显式记录为假设，并给出后续验证路径和解除条件。 |
| architecture_vision | 第一性原理 | `G202` | 在边界不清或约束混乱时，将事实、假设和推导拆开，从最基础的已知条件重新构建问题定义。 |
| architecture_vision | 事件风暴 | `G202` | 在业务域复杂、事件链长的场景下，用于补充识别关键业务事件和跨角色交互，不作为默认主方法。 |
| technical_strategy | 架构原则约束映射 | `G201` | 将 `G202` 已冻结的架构原则映射到技术路线、选型和实施约束，防止策略漂移。 |
| technical_strategy | 约束驱动选型 | `G201` | 基于业务、技术、合规、资源和交付约束选择候选方案，要求每个推荐结论都能回扣约束来源。 |
| technical_strategy | 技术域分层分析 | `G201` | 将技术策略拆成应用架构、数据与集成、运行与部署等技术域分别分析，避免把不同层问题混在一起。 |
| technical_strategy | 决策矩阵 | `G201` | 对多个候选方案按统一维度进行比较，形成可复核的推荐与备选结论。 |
| technical_strategy | ATAM 权衡分析 | `G201` | 以质量属性冲突和权衡为中心评估方案优劣，支撑技术路线和关键选型决策。 |
| technical_strategy | 风险驱动决策 | `G201` | 以主要风险和失败代价作为决策依据，优先选择风险可控、验证路径清晰的策略。 |
| technical_strategy | 成本-收益-风险权衡 | `G201` | 在性能、安全、成本、交付速度等多维度之间做显式取舍，避免只有单一推荐而缺少代价说明。 |
| technical_strategy | ADR 思维 | `G201` | 按“背景、候选、决策、后果”组织关键策略结论，为后续 ADR 正式化做准备。 |
| technical_strategy | 替代路线与触发条件分析 | `G201` | 除推荐方案外，明确 fallback 路线以及何时需要切换，降低后续蓝图设计和实施风险。 |
| technical_strategy | 假设清单与验证计划 | `G201` | 对策略中的前提假设、待验证点和试验路径做结构化记录，避免未验证假设直接固化成设计约束。 |
| architecture_blueprint | 蓝图范围切片 | `G203` | 基于技术策略和阶段边界，明确哪些结构内容进入当前蓝图、哪些下沉到详细设计或后续阶段，避免蓝图过宽或过浅。 |
| architecture_blueprint | 架构视图映射 | `G203` | 以上下文、模块、部署、集成等视图组织蓝图，确保每类视图都有明确目标、覆盖对象和相互映射关系。 |
| architecture_blueprint | 约束落图 | `G203` | 将上游策略约束、质量属性和部署边界显式映射到蓝图视图、组件结构和运行拓扑中，避免约束只停留在文字层。 |
| architecture_blueprint | 组件职责分解 | `G203` | 将系统能力拆分到核心组件或模块，明确职责边界、所有权和依赖方向，为后续详细设计提供结构骨架。 |
| architecture_blueprint | 接口与依赖建模 | `G203` | 对关键接口、依赖类型、调用方向和异常处理路径做结构化建模，降低后续实现阶段的歧义。 |
| architecture_blueprint | 关键链路走查 | `G203` | 以关键业务链路或高风险链路为主线，验证蓝图结构、组件职责和接口方向是否自洽。 |
| architecture_blueprint | 部署拓扑建模 | `G203` | 描述运行节点、部署单元、环境边界与关键非功能约束，确保蓝图能落到实际运行形态。 |
| architecture_blueprint | ADR 固化 | `G203` | 将关键架构决策沉淀为正式 ADR，并建立与视图、组件、约束之间的索引关系，便于评审和演进。 |
| architecture_blueprint | 蓝图追溯映射 | `G203` | 将愿景、技术策略、蓝图视图、组件、交互链路、部署拓扑与 ADR 建立显式回链，确保下游评审和后续设计可以稳定追溯。 |
| architecture_blueprint | 视图一致性检查 | `G203` | 检查结构视图、交互视图、部署视图与 ADR 之间是否相互一致，避免不同视图各自成立但整体不自洽。 |
| architecture_blueprint | 风险热点复核 | `G203` | 针对蓝图中最可能引发评审争议或实施风险的热点区域做重点复核，提前暴露不稳定边界。 |
| architecture_review | 系统上下文图 | `G204` | 用系统边界与关键交互重新校准当前评审范围，避免把不属于本轮 architecture 文档包的内容混入评审结论。 |
| architecture_review | 架构原则约束映射 | `G204` | 将愿景、策略与蓝图中的原则和约束逐项映射到评审检查点，识别漂移、缺口和未闭环事项。 |
| architecture_review | 约束分层法 | `G204` | 按业务、技术、合规、交付、组织等层次归类评审发现，避免把不同层级问题混成单一结论。 |
| architecture_review | 技术域分层分析 | `G204` | 从应用结构、数据与集成、运行部署等技术域验证蓝图和 ADR 的承接完整性。 |
| architecture_review | ADR 思维 | `G204` | 按“背景、候选、决策、后果”方式复核 ADR 与蓝图是否一致，并为返工动作提供清晰依据。 |
| architecture_review | 蓝图追溯映射 | `G204` | 以 `G203` 的结构化最小字段为对象，检查愿景、技术策略、蓝图元素与 ADR 之间的追溯链是否完整、稳定、可审计。 |
| architecture_review | 质量属性场景 | `G204` | 用质量场景验证蓝图是否真实响应关键性能、可靠性、安全和扩展性要求。 |
| architecture_review | ATAM 权衡分析 | `G204` | 从质量属性权衡和设计取舍角度验证当前蓝图与 ADR 的合理性，而不是只看静态一致性。 |
| architecture_review | 风险驱动分析 | `G204` | 以当前 architecture 文档包中的高风险项为主线组织评审，优先暴露最可能影响阶段交付的缺口。 |
| architecture_review | 视图一致性检查 | `G204` | 从评审角度检查结构、交互、部署视图与 ADR、关键约束之间是否保持一致，避免蓝图多视图各自成立但整体冲突。 |
| architecture_review | 风险驱动决策 | `G204` | 用于收敛评审结论和返工优先级，明确哪些问题必须在进入质量门前先处理。 |
| architecture_review | 假设清单与验证计划 | `G204` | 将尚未证实的判断留在假设清单中，并为后续质量门或返工提供明确验证路径。 |
| architecture_review | 替代路线与触发条件分析 | `G204` | 对争议较大的设计结论给出可接受替代路线以及触发切换的条件，便于评审闭环。 |

## 使用建议

1. `G202` 优先使用 `architecture_vision` 类方法。
2. `G201` 优先使用 `technical_strategy` 类方法。
3. `G203` 优先使用 `architecture_blueprint` 类方法。
4. `G204` 优先使用 `architecture_review` 类方法。
5. 若某步骤使用了可选方法，应在对应 `template.md` 的“方法检查清单”中记录实际使用情况。
6. `G202/G201/G203/G204` 中出现的方法名，应以本目录为唯一标准引用来源。
