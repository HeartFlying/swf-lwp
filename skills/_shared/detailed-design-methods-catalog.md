# Detailed Design Methods Catalog

说明：

1. 本文档使用“方法名称 + 适用对象 + 简要说明”组织方式，沉淀 detailed_design 阶段的标准方法目录。
2. 当前优先覆盖 `G301`（组件详细设计），并为后续 `G302`（接口详细设计）、`G303`（数据详细设计）预留共用方法。
3. `G301/G302/G303` 中出现的方法名，应优先以本目录为唯一标准引用来源；具体在哪个步骤使用，以各 SKILL 的 `SKILL.md`、`template.md` 和 `references/execution-details.md` 为准。

| category | technique_name | applicable_skill | description |
|---|---|---|---|
| detailed_design_shared | 设计范围冻结 | `G301`、`G302`、`G303` | 在进入详细设计正文前，先冻结当前轮设计边界、延后项和不在本轮定稿的内容，避免把组件、接口、数据设计混写。 |
| detailed_design_shared | 约束回链 | `G301`、`G302`、`G303` | 将架构蓝图、技术策略、评审结论和澄清结果回链到当前设计项，确保详细设计不脱离上游约束。 |
| detailed_design_shared | 契约驱动设计 | `G301`、`G302`、`G303` | 先明确边界责任、输入输出、消费语义和待细化项，再展开内部实现细节，避免后续交接字段不稳定。 |
| detailed_design_shared | 失败模式分析 | `G301`、`G302`、`G303` | 从失败场景、检测方式、影响范围和恢复路径出发设计异常处理，而不是只列出报错分支。 |
| detailed_design_shared | 可测试性分层设计 | `G301`、`G302`、`G303` | 将测试设计拆成单元、组件、集成、契约和评审验证等层次，明确观察点、替身需求和验收方式。 |
| detailed_design_shared | 可观测性设计 | `G301`、`G302`、`G303` | 设计日志、指标、追踪、告警和诊断上下文，确保关键成功路径和失败路径可观测。 |
| component_design | 职责-协作映射 | `G301` | 将组件职责、协作关系、输入输出和责任边界放入统一映射表，避免只给组件列表而没有协作语义。 |
| component_design | 内部结构分解 | `G301` | 将组件细化为内部模块、层次和扩展点，并明确内部职责分工与依赖方向。 |
| component_design | 依赖反转校验 | `G301` | 检查关键依赖是否满足既定边界和抽象层次，避免低层能力反向污染高层设计。 |
| component_design | 时序建模 | `G301` | 对关键交互和流程按时序展开，明确调用顺序、同步异步边界、状态变化和协作触发点。 |
| component_design | 状态机建模 | `G301` | 对存在状态变化的组件建立状态对象、状态迁移、触发条件和失败行为模型。 |
| component_design | 规则决策表 | `G301` | 用结构化决策表表达关键规则、分支条件和处理结果，降低规则歧义。 |
| component_design | 前置/后置条件建模 | `G301` | 为关键操作、状态迁移和规则执行定义前置条件、后置条件与不满足时的行为。 |
| component_design | 并发与一致性分析 | `G301` | 分析并发访问、重试、幂等、顺序性和一致性要求，避免组件设计只覆盖单线程理想路径。 |
| component_design | 接口数据边界对齐 | `G301`、`G302` | 将组件内部设计与后续接口/数据详细设计的交接边界对齐，明确哪些内容已定稿、哪些内容留给下游细化；在接口详细设计中继续用于对齐组件交接边界与接口契约边界。 |
| component_design | 风险热点预判 | `G301` | 在复杂组件或高风险链路上预判最可能引发返工的热点区域，提前补充约束或待确认项。 |
| component_design | 复杂度热点分析 | `G301` | 对高复杂度算法、状态机或协作链路进行重点分析，提前暴露性能和可维护性风险。 |
| interface_design | 契约一致性校验 | `G302` | 对接口请求、响应、错误语义、版本和幂等约束做一致性校验。 |
| interface_design | 兼容性策略设计 | `G302` | 对接口版本演进、向后兼容、灰度切换、弃用策略和兼容窗口做结构化设计。 |
| interface_design | 幂等与重试语义设计 | `G302` | 对幂等键、重复请求、超时重试、回调重放和去重窗口做明确语义设计，避免调用链在失败场景下失稳。 |
| interface_design | 时序交互建模 | `G302` | 对关键接口调用链、异步回调、超时和重试流程做时序建模。 |
| interface_design | 错误语义分层 | `G302` | 将错误码、失败语义、补偿建议和调用方责任做分层定义。 |
| data_design | 数据对象建模 | `G303` | 对核心数据对象、属性、关系和生命周期做结构化建模。 |
| data_design | 读写边界分析 | `G303` | 明确所有权、写入责任、读取视图和跨组件共享边界。 |
| data_design | 一致性与演化设计 | `G303` | 设计一致性级别、演化策略、索引与迁移风险，确保数据方案可持续演进。 |
| data_design | 迁移切换策略设计 | `G303` | 对双写双读、回填校验、切换窗口、回滚条件和切换门槛做结构化设计，避免数据迁移只停留在概念层。 |
| data_design | 数据保留与归档策略设计 | `G303` | 对保留周期、归档策略、删除条件、恢复窗口和审计要求做结构化设计，确保生命周期治理可执行。 |

## 使用建议

1. `G301` 优先使用 `component_design` 与 `detailed_design_shared` 类方法。
2. `G302` 优先使用 `interface_design` 与 `detailed_design_shared` 类方法。
3. `G303` 优先使用 `data_design` 与 `detailed_design_shared` 类方法。
4. 若某方法被写入步骤、模板或执行细则，应使用本目录中的标准名称，不使用同义词和临时命名。
