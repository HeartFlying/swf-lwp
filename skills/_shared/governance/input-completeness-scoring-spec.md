# 信息完整度评分规范

## 1. 目的

本文档将“信息完整度评分”固化为可重复执行的工程规范，用于 requirements 阶段入口在进入主流程前进行预检。

本文档解决以下问题：

1. 评分字段如何落到固定检查表。
2. 每个字段如何打分。
3. 四个维度和总分如何计算。
4. 何时触发澄清。
5. 何时进入 Fast、Standard、Complete。

---

## 2. 输出要求

评分过程必须输出以下 5 类结果：

1. `dimension_scores`：四个维度分值
2. `total_score`：总分
3. `missing_fields`：缺失字段清单
4. `clarification_required`：是否必须澄清
5. `mode_decision_basis`：模式判定依据

建议输出路径：

- 需求接收阶段：并入 `001-requirements-intake.md`
- 需要澄清时：同步写入 `001a-requirements-clarification.md`

---

## 3. 评分维度与权重

| 维度 | 代码 | 权重 | 目标 |
|---|---|---|---|
| 需求清晰度 | D1 | 30% | 判断目标、功能、验收是否清晰 |
| 边界完整性 | D2 | 25% | 判断用户、场景、范围边界是否清晰 |
| 技术约束 | D3 | 25% | 判断技术栈、部署、性能、安全是否清晰 |
| 业务背景 | D4 | 20% | 判断业务目标、价值、优先级、时间预期是否清晰 |

约束：

1. 所有维度统一换算到 0-100。
2. 总分统一四舍五入到整数。
3. 任一字段都必须有可审计打分依据。

---

## 4. 字段检查表

### 4.1 D1 需求清晰度

| 字段代码 | 字段名称 | 判定说明 | 是否关键字段 |
|---|---|---|---|
| D1-F1 | 目标描述 | 是否说明要解决什么问题、达成什么结果 | 是 |
| D1-F2 | 核心功能列表 | 是否列出主要功能或能力 | 是 |
| D1-F3 | 验收标准 | 是否说明如何判断做完 | 是 |
| D1-F4 | 成功指标 | 是否有可衡量结果或业务成功标准 | 否 |

### 4.2 D2 边界完整性

| 字段代码 | 字段名称 | 判定说明 | 是否关键字段 |
|---|---|---|---|
| D2-F1 | 目标用户 | 是否说明服务对象或用户角色 | 是 |
| D2-F2 | 核心场景 | 是否说明典型使用场景 | 是 |
| D2-F3 | 范围边界 | 是否明确在做什么 | 是 |
| D2-F4 | 不做项 | 是否明确 out of scope | 是 |

### 4.3 D3 技术约束

| 字段代码 | 字段名称 | 判定说明 | 是否关键字段 |
|---|---|---|---|
| D3-F1 | 技术栈偏好或限制 | 是否说明偏好、禁用项或已有技术栈 | 否 |
| D3-F2 | 部署环境 | 是否说明本地、云、私有化等部署要求 | 是 |
| D3-F3 | 性能要求 | 是否说明响应时间、吞吐或容量要求 | 是 |
| D3-F4 | 安全合规要求 | 是否说明权限、审计、合规或数据安全要求 | 是 |

### 4.4 D4 业务背景

| 字段代码 | 字段名称 | 判定说明 | 是否关键字段 |
|---|---|---|---|
| D4-F1 | 业务背景 | 是否说明项目背景和上下文 | 否 |
| D4-F2 | 业务价值 | 是否说明价值、收益或必要性 | 否 |
| D4-F3 | 优先级 | 是否说明轻重缓急 | 是 |
| D4-F4 | 时间预期 | 是否说明交付时间、节奏或时效要求 | 否 |

---

## 5. 字段打分规则

每个字段只能取以下三个值：

| 分值 | 含义 | 判定标准 |
|---|---|---|
| `1` | 明确 | 有具体、可执行、可落地描述 |
| `0.5` | 部分明确 | 有描述，但不完整、不可直接执行或明显模糊 |
| `0` | 缺失 | 没有提供，或几乎无法据此建模 |

判定示例：

| 情况 | 分值 |
|---|---|
| “系统响应时间小于 200ms” | `1` |
| “系统要快一些” | `0.5` |
| 完全未提性能要求 | `0` |

补充规则：

1. 相同字段不得同时打多个分值。
2. 无法判断时，默认从严按 `0.5` 或 `0` 处理，不按 `1` 放宽。
3. 若字段内容彼此冲突，先记入 `missing_fields` 和风险，再按较低分值处理。

---

## 6. 计算公式

设每个维度有 4 个字段，字段分值为 `x_ij`：

- 维度得分：
  - `dimension_score_i = (sum(x_ij) / 4) * 100`
- 总分：
  - `total_score = 0.30*D1 + 0.25*D2 + 0.25*D3 + 0.20*D4`

输出要求：

1. `dimension_score_i` 四舍五入到整数。
2. `total_score` 四舍五入到整数。
3. 原始字段打分明细必须保留，不能只保留总分。

---

## 7. 缺失字段识别规则

### 7.1 缺失字段定义

以下字段需写入 `missing_fields`：

1. 评分为 `0` 的字段
2. 属于关键字段且评分为 `0.5` 的字段
3. 内容冲突导致无法直接用于建模的字段

### 7.2 缺失字段输出结构

每个缺失字段至少包含：

- `field_code`
- `field_name`
- `dimension`
- `score`
- `reason`
- `impact`
- `requires_clarification`

示例：

```yaml
missing_fields:
  - field_code: D2-F3
    field_name: 范围边界
    dimension: D2
    score: 0
    reason: 未明确本次交付范围
    impact: 无法冻结需求基线
    requires_clarification: true
```

---

## 8. 澄清触发规则

### 8.1 必须触发澄清

满足以下任一条件，必须启动澄清：

1. `total_score < 60`
2. `60 <= total_score < 90` 且 D1 或 D2 存在关键字段缺失
3. 连续评分后仍存在“无法冻结基线”的关键缺失

### 8.2 可直接进入主流程

满足以下条件可不额外澄清直接进入模式判定：

1. `total_score >= 90`
2. D1 不低于 50
3. D2、D3 不存在“关键字段全缺失”

### 8.3 澄清轮次限制

1. 每轮最多 3-5 个问题
2. 优先覆盖关键字段
3. 最多连续 3 轮
4. 超过 2-3 轮仍缺关键项时，必须记录假设、待确认项和风险，并按 Complete 继续

---

## 9. 澄清问题生成规则

生成澄清问题时必须遵循以下顺序：

1. 先问影响需求基线冻结的问题
2. 再问影响模式判定的问题
3. 最后问影响分析深度的问题

问题形式要求：

1. 优先封闭式问题
2. 每题提供 2-4 个推荐选项
3. 必须允许用户补充自定义答案
4. 每个选项应附简短适用说明

推荐澄清主题：

- 目标与成功标准
- 目标用户与核心场景
- 范围边界与不做项
- 技术约束与部署环境
- 优先级与时间预期

---

## 10. 模式判定规则

优先级从高到低执行：

1. 用户显式指定 `complete`：进入 Complete
2. 用户显式指定 `fast` 且 `total_score >= 90` 且 `D1 >= 50`：进入 Fast
3. 若 `D2 < 25` 或 `D3 < 25`：进入 Complete
4. 若 `total_score < 60`：进入 Complete
5. 若 `60 <= total_score < 90`：进入 Standard
6. 若 `total_score >= 90`：
   - 用户明确要求快速：Fast
   - 其他情况：Standard

补充约束：

1. `D1 < 50` 时禁止 Fast。
2. 用户指定模式与规则冲突时，必须输出冲突说明。
3. 冲突未确认前，不得推进正式 Skill 执行。

---

## 11. 标准输出格式

建议统一输出为以下结构：

```yaml
input_completeness_assessment:
  dimension_scores:
    D1: 75
    D2: 63
    D3: 50
    D4: 88
  total_score: 68
  missing_fields:
    - field_code: D2-F4
      field_name: 不做项
      score: 0
      reason: 未说明 out of scope
      impact: 范围边界不稳定
      requires_clarification: true
  clarification_required: true
  clarification_reason:
    - D1_or_D2_key_field_missing
  recommended_mode: standard
  final_mode: standard
  mode_decision_basis: score_60_89
```

---

## 12. 验收标准

本文档通过条件：

1. 字段表完整覆盖 D1-D4。
2. 每个字段都能按 `0/0.5/1` 复算。
3. 澄清触发规则和模式判定规则可直接执行。
4. 输出格式可被 requirements 阶段入口 引用。
