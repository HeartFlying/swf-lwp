# 质量检查器清单

## 1. 目的

本文档定义工作流中自动化质量检查器的最小需求清单，作为后续 `GS-Quality-Check` 与脚本实现的输入。

本文档解决以下问题：

1. 需要实现哪些检查器。
2. 每个检查器检查什么。
3. 每个检查器的输入、输出和失败级别是什么。
4. 汇总执行器如何组织单项检查。
5. 哪些检查属于必选，哪些可作为扩展。

---

## 2. 适用范围

本清单适用于以下正式文档：

- `governance/*.md`
- `skills/*/SKILL.md`
- `skills/*/template.md`
- `references/*.md`
- 阶段产物和评审报告

本清单不直接约束旧实现目录和 `WORKFLOW.md`，但旧实现可作为参考来源。

---

## 3. 设计原则

1. 自动化优先：能脚本化的检查项不依赖人工。
2. 分层检查：单文件检查、跨文档检查、阶段级检查分开执行。
3. 可追溯：每条失败必须能定位到文件、规则和原因。
4. 可汇总：所有单项检查器结果必须被统一执行器收口。
5. 可扩展：后续新增规则时，不破坏现有输出格式。

---

## 4. 检查器清单总览

| 检查器 ID | 检查器名称 | 类型 | 是否必选 | 核心目标 |
|---|---|---|---|---|
| QC-001 | 文件存在性检查器 | 单文件/批量 | 是 | 检查目标文件是否存在 |
| QC-002 | Markdown 格式检查器 | 单文件 | 是 | 检查标题、列表、表格、代码块等结构 |
| QC-003 | 文档完整性检查器 | 单文件 | 是 | 检查必需章节和内容完整性 |
| QC-004 | 规模限制检查器 | 单文件 | 是 | 检查 10000 行限制和拆分规则 |
| QC-005 | 自包含性检查器 | 单文件/跨文件 | 是 | 检查引用是否存在、内容是否可独立理解 |
| QC-006 | 一致性检查器 | 跨文档 | 是 | 检查术语、编号、关键数据一致性 |
| QC-007 | 追溯性检查器 | 跨文档 | 是 | 检查需求到设计的追溯关系 |
| QC-008 | 图表检查器 | 单文件 | 可选增强 | 检查图表语法、引用和完整性 |
| QC-009 | 命名与路径检查器 | 单文件/批量 | 是 | 检查目录结构、命名规则、正式路径 |
| QC-010 | 台账关联检查器 | 跨文档 | 是 | 检查任务台账、证据路径和正式文档关联 |
| QC-011 | 汇总执行器 | 汇总器 | 是 | 统一执行检查并产出汇总报告 |

## 5. 单项检查器需求

### 5.1 QC-001 文件存在性检查器

检查目标：

1. 目标文档必须存在。
2. 目标路径必须位于正式目录或正式产物目录。
3. 依赖文件声明存在时，必须能被定位。

输入：

- 待检查文件路径列表
- 可选依赖文件列表

输出：

- `exists: true/false`
- 缺失文件列表
- 错误级别

失败级别：

- 文件缺失：`critical`

### 5.2 QC-002 Markdown 格式检查器

检查目标：

1. 一级标题数量正确
2. 标题层级连续
3. 列表语法合法
4. 表格结构合法
5. 代码块成对闭合
6. 链接语法合法

输入：

- 单个 Markdown 文件

输出：

- 格式得分
- 违规项列表
- 行号或位置

失败级别：

- 严重结构错误：`critical`
- 一般格式问题：`major`

### 5.3 QC-003 文档完整性检查器

检查目标：

1. 必需章节是否存在
2. 章节是否为空
3. 段落是否明显截断
4. 表格、代码块、清单是否完整

输入：

- 单个 Markdown 文件
- 对应文档类型的必需章节模板

输出：

- 完整性得分
- 缺失章节列表
- 空章节列表

失败级别：

- 缺失必需章节：`critical`
- 章节内容不足：`major`

### 5.4 QC-004 规模限制检查器

检查目标：

1. 单文件是否超过 10000 行
2. 超限时是否已正确拆分
3. 拆分后各文件是否自包含

输入：

- 单个文件或文档主入口

输出：

- 行数
- 是否超限
- 是否满足拆分规则

失败级别：

- 超限且未拆分：`critical`
- 已拆分但不完整：`major`

### 5.5 QC-005 自包含性检查器

检查目标：

1. 本文档引用的相对路径存在
2. 文档不依赖外部隐含上下文才能理解
3. 拆分主文件能正确索引子文件

输入：

- 单个 Markdown 文件

输出：

- 引用检查结果
- 断链列表
- 自包含结论

失败级别：

- 断链：`critical`
- 主文件索引不完整：`major`

### 5.6 QC-006 一致性检查器

检查目标：

1. 术语命名一致
2. Skill 编号、阶段编号、任务编号一致
3. 模式名、状态名、路径名一致
4. 关键阈值、权重、规则不冲突

输入：

- 一组相关文档

输出：

- 不一致项列表
- 涉及文件
- 建议主值

失败级别：

- 核心规则冲突：`critical`
- 一般术语不一致：`major`

### 5.7 QC-007 追溯性检查器

检查目标：

1. 需求到架构是否可追溯
2. 架构到详细设计是否可追溯
3. 任务到证据是否可追溯
4. 文档到任务编号是否可追溯

输入：

- 需求、架构、设计和台账相关文档

输出：

- 追溯矩阵
- 追溯覆盖率
- 缺失链路列表

失败级别：

- 核心链路缺失：`critical`
- 覆盖率不足：`major`

### 5.8 QC-008 图表检查器

检查目标：

1. 图表文件或图表块语法可解析
2. 图表引用存在
3. 图表标题与正文引用一致

输入：

- 含图表的文档

输出：

- 图表检查结果
- 图表错误列表

失败级别：

- 图表语法错误：`major`

备注：

- 当前阶段可作为增强项，不要求先实现全部细节。

### 5.9 QC-009 命名与路径检查器

检查目标：

1. 正式文档是否写入项目根目录下标准结构
2. 文件命名是否符合约定
3. Skill 目录结构是否完整
4. 旧目录是否被误用为正式输出目录

输入：

- 仓库内相关路径

输出：

- 命名违规项
- 路径违规项

失败级别：

- 写入错误目录：`critical`
- 命名不符合规范：`major`

### 5.10 QC-010 台账关联检查器

检查目标：

1. 每个已完成任务是否存在证据目录
2. `evidence_path` 是否存在
3. 每个已完成任务的证据目录是否非空（至少包含 1 份过程证据文件）
4. `resume_from` 是否在非完成态任务中存在
5. 正式文档是否能回溯到任务台账

输入：

- 证据目录
- 正式文档目录

输出：

- 台账关联检查结果
- 缺失证据项
- 证据为空的任务项
- 状态字段异常项

失败级别：

- 已完成任务无证据目录：`critical`
- 已完成任务证据目录为空（无过程证据文件）：`major`
- 执行中任务无恢复点：`major`

## 6. 汇总执行器需求

`QC-011` 作为统一执行器，至少满足以下能力：

1. 接收单文件或目录作为输入
2. 根据文档类型选择应执行的检查器
3. 汇总所有检查器结果
4. 产出统一 Markdown 报告
5. 输出总状态：`pass / pass_with_warning / fail`

建议输入参数：

- `--input`
- `--mode`：`single / batch / stage`
- `--output`
- `--config`

建议输出结构：

```yaml
quality_check_summary:
  target: skills/g100-requirements-entry/SKILL.md
  overall_status: fail
  executed_checkers:
    - QC-001
    - QC-002
    - QC-003
  scores:
    markdown_format: 92
    completeness: 88
  issues:
    - checker: QC-003
      severity: critical
      message: 缺少“输出”章节
```

---

## 8. 失败级别定义

| 级别 | 含义 | 处理规则 |
|---|---|---|
| `critical` | 阻止任务完成的严重问题 | 不得进入用户评审或阶段推进 |
| `major` | 需要修复的重要问题 | 原则上需修复后再提交评审 |
| `minor` | 不阻塞推进的问题 | 可记录后继续 |
| `warning` | 提示性问题 | 不阻塞推进 |

---

## 9. 验收标准

本文档通过条件：

1. 覆盖完整性、规范性、一致性、追溯性、路径和台账关联检查。
2. 每个检查器都有输入、输出和失败级别说明。
3. 定义统一执行器和最小检查矩阵。
4. 能作为 T104 和后续 `GS-Quality-Check` 的输入。
