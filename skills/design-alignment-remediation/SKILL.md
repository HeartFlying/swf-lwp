# Design Alignment Remediation（设计对齐修复）

## 目标

将已识别的设计漂移项系统性地修复，确保跨技术栈（proto/SQL/Go/TS/C++）的代码与设计规格重新一致。核心原则：

1. **先计划后执行**：修复开始前必须有详细计划，每个修复项必有验收标准
2. **用户确认门**：计划制定后、执行前，必须经用户确认
3. **多子代理并行**：无文件冲突的修复项并行派发，有依赖的串行
4. **全栈验收**：修复完成后跨技术栈验证，不止单点检查

## 输入

接受以下任一来源的漂移清单：

- **默认来源**：`design-drift-audit` Skill 产出的漂移审计报告
- **用户指定**：用户手动提供的漂移项清单（需满足最低信息要求：每个漂移项必须包含"设计状态"和"代码状态"）

若用户未指定输入且无审计报告存在，先询问用户提供漂移清单，不可凭空执行。

---

## 阶段一：消费审计报告 → 生成详细修复计划

### 1.1 解析漂移项

对审计报告中的每个漂移项，提取并补全以下信息：

```
逐项解析:
  - 漂移项ID (如无则分配: DRIFT-01, DRIFT-02, ...)
  - 漂移描述: 设计定义 vs 代码现状的差距
  - 涉及技术栈: proto / SQL / Go / TypeScript / C++ (可多选)
  - 涉及文件: 精确到文件路径
  - 变更类型: 新增 / 修改 / 删除
  - "当前→目标": 精确到字段级别的前后对比
  - 影响评估: 不修复会阻塞什么下游工作
```

补全规则：
- 若审计报告已包含"当前状态→目标状态"的代码级定义，直接引用
- 若审计报告只描述了差距但未给出目标代码，则读取设计文档补全目标定义
- "涉及文件"必须精确到文件路径，不可写"相关文件若干"

### 1.2 依赖排序

按以下规则确定修复项的执行顺序：

```
铁律: contracts(proto/openapi) 永远是第一组
原因: proto/openapi 是跨语言单一事实来源，所有下游技术栈依赖它

排序规则:
  1. contracts 变更 (proto + openapi) → 最高优先级
  2. 数据库变更 (SQL DDL + Go model struct) → 依赖 contracts 完成
  3. 后端变更 (Go services) → 依赖 contracts + 数据库完成
  4. 前端变更 (TypeScript) → 依赖 contracts 完成
  5. 算法变更 (C++) → 依赖 contracts 完成

同级技术栈内的排序:
  - 先改类型定义/结构体 → 再改业务逻辑
  - 先改接口定义 → 再改实现
```

### 1.3 并行分组

将修复项分配到并行组。同组内的修复项**无文件冲突、无依赖关系**，可并行执行。

分组规则：
- **按技术栈分组**：同一技术栈内的多个修复项，若涉及文件无交集，归入同一并行组
- **按修复模块分组**：若一个修复项跨多个技术栈（如"TrajectoryPoint 字段变更"涉及 proto+SQL+Go+TS+C++），则拆分为每个技术栈一个子任务，分别归入各自技术栈的并行组
- **一组完成后方可启动下一组**（组间串行，组内并行）

示例分组结构：
```
Group 1 (contracts, 优先级最高):
  - 子任务1.1: proto 字段变更
  - 子任务1.2: openapi schema 新增

Group 2 (database, 依赖 Group 1):
  - 子任务2.1: SQL DDL 变更 + Go model struct 同步

Group 3 (backend + frontend + algorithm, 依赖 Group 1+2, 组内并行):
  - 子任务3.1: Go WebSocket message struct 更新
  - 子任务3.2: TypeScript 类型定义更新
  - 子任务3.3: C++ struct + 算法逻辑更新
```

### 1.4 为每个修复项生成验收标准

每个修复项至少包含以下验收标准：

```
必选验收项:
  AC-编译: 该技术栈编译/构建零 error
  AC-一致性: 变更字段在相关技术栈中存在且类型一致 (grep 验证)

可选验收项 (根据变更类型):
  AC-测试: 若涉及逻辑变更，相关测试通过
  AC-契约: 若涉及 proto/openapi，生成代码零 diff
  AC-安全: 若涉及认证/授权，安全扫描通过
```

验收标准必须是**机械可验证的**，格式：`工具/命令 + 预期结果`。禁止"逻辑正确"、"代码整洁"等主观描述。

### 1.5 输出修复计划

按以下模板输出，**必须包含用户确认提示**：

```markdown
# 设计对齐修复计划

## 修复总览
- 漂移项总数: N
- 涉及文件: M (新增 X, 修改 Y, 生成 Z)
- 并行组数: P
- 预估影响范围: [哪些下游模块会受影响]

## 执行顺序

Group 1 → Group 2 → Group 3
(每组内的子任务并行执行，组间串行)

## 各并行组详情

### Group 1: [名称] (优先级: 最高)

| 子任务ID | 漂移项 | 技术栈 | 涉及文件 | 验收标准 |
|----------|--------|--------|----------|----------|
| G1-T1 | DRIFT-01 描述 | proto | contracts/proto/x.proto | AC-编译, AC-契约 |
| G1-T2 | DRIFT-02 描述 | openapi | contracts/openapi/x.yaml | AC-编译, AC-一致性 |

### Group 2: [名称] (依赖 Group 1)
...

### Group 3: [名称] (依赖 Group 1+2)
...

## 验收标准矩阵

| 子任务ID | AC-编译 | AC-一致性 | AC-测试 | AC-契约 | AC-安全 |
|----------|---------|-----------|---------|---------|---------|
| G1-T1 | protoc 零 error | - | - | protoc-gen.sh 零 diff | - |
| G2-T1 | go build 通过 | grep pile_number .go 存在 | go test 通过 | - | - |
...

## 风险提示
- [列出需注意的兼容性风险，如 proto field number 变更]

---
> 请确认以上修复计划。你可以：
> - 回复"确认"或"开始执行" → 进入阶段三并行修复
> - 移除某个修复项 → 告知要跳过的子任务ID
> - 调整顺序 → 告知期望的执行顺序
> - 补充验收标准 → 告知要追加的验收项
```

**此阶段禁止直接开始修改代码。必须等待用户确认。**

---

## 阶段二：用户确认（交互点）

呈现修复计划后，等待用户操作。用户可能的响应及处理方式：

| 用户操作 | 处理 |
|----------|------|
| 确认执行 | 进入阶段三 |
| 跳过某修复项 | 从计划中移除，重新展示受影响的任务数，确认后进入阶段三 |
| 调整顺序 | 重新排列并行组，确认后进入阶段三 |
| 补充验收标准 | 追加到对应子任务，确认后进入阶段三 |
| 放弃 | 终止流程，不执行任何修改 |

若用户确认时未指定其他执行策略，默认按计划顺序执行。

---

## 阶段三：多子代理并行修复

### 3.1 派发策略

按并行组顺序派发：

```
for each Group in 计划.并行组 (按优先级从高到低):
  1. 并行启动 Group 内所有子任务的子代理
  2. 等待该组所有子代理完成
  3. 检查结果:
     - 全部 PASS → 进入下一 Group
     - 任一 FAIL → 立即暂停，报告用户，不执行后续 Group
```

### 3.2 子代理任务模板

每个子代理接收以下结构化任务：

```
## 子任务: {子任务ID}

### 修复目标
{漂移项描述}

### 当前→目标
当前状态: {代码现状}
目标状态: {设计定义}

### 涉及文件
- {文件路径1} — {变更类型}
- {文件路径2} — {变更类型}

### 验收标准 (逐条机械执行)
1. {AC-1 命令 + 预期结果}
2. {AC-2 命令 + 预期结果}

### 约束
- 只改涉及文件清单中的文件，不顺手改其他文件
- 变更完成后立即运行验收标准，记录实际输出
- 若验收失败，回报失败原因 (不自行修复)

### 回报格式
完成后返回:
- status: PASS / FAIL
- 变更文件: [实际修改的文件列表]
- 验收结果:
  1. {AC-1}: PASS/FAIL — {实际输出摘要}
  2. {AC-2}: PASS/FAIL — {实际输出摘要}
- 备注: {如遇问题，描述具体错误}
```

### 3.3 失败处理

```
子代理返回 FAIL 时:
  1. 立即暂停当前组及所有后续组
  2. 向用户报告:
     - 哪个子任务失败
     - 哪个验收标准未通过
     - 实际的错误输出
     - 建议的修复方向 (但不自动修复)
  3. 等待用户指示:
     - 修复后重试当前子任务
     - 跳过该修复项继续
     - 放弃全部修复
```

## 阶段四：汇总 + 全栈验收

所有并行组完成后，执行全栈验收。

### 4.1 逐修复项汇总

| 子任务ID | 漂移项 | 状态 | 变更文件数 | 备注 |
|----------|--------|------|-----------|------|
| G1-T1 | DRIFT-01 | PASS | 1 | proto 字段变更完成 |
| ... | ... | ... | ... | ... |

### 4.2 跨技术栈全栈验收

执行以下验证（复用项目 `acceptance-criteria` 规则的八大维度框架）：

| 维度 | 验收方法 | 合格门槛 |
|------|----------|----------|
| D8 契约一致性 | `protoc-gen.sh` + `git diff contracts/proto/gen/` | 零 diff |
| D8 字段全链路对齐 | `grep -r "<关键字段>" --include="*.proto" --include="*.sql" --include="*.go" --include="*.ts" --include="*.h" --include="*.cpp"` | 所有技术栈均含目标字段 |
| D4 编译零 Warning | `go vet ./...` + `vue-tsc --noEmit` + `cmake --build` | 0 warning |
| D1 测试覆盖率 | `go test ./... -coverprofile=coverage.out` | 整体 ≥ 60% |

若涉及其他维度（D2 性能、D3 安全、D5 集成断言、D6 数据完整性、D7 CI），按需追加。

### 4.3 输出修复报告

```markdown
# 设计对齐修复报告

## 修复概要
- 执行日期: YYYY-MM-DD
- 漂移项总数: N | 已修复: N | 跳过: 0 | 失败: 0
- 涉及文件: M (新增 X, 修改 Y, 生成 Z)
- 并行组数: P | 子任务总数: T

## 逐修复项详情

### DRIFT-01: {漂移项描述}
- 状态: ✅ PASS
- 子任务: G1-T1
- 变更文件:
  - `contracts/proto/x.proto` — 修改: field 3 double x → int64 pile_number
- 验收结果:
  - AC-编译: PASS (protoc 编译通过)
  - AC-契约: PASS (protoc-gen.sh 生成零 diff)

### DRIFT-02: {漂移项描述}
- 状态: ✅ PASS
...

## 全栈验收结果

| 维度 | 状态 | 说明 |
|------|------|------|
| D8 契约一致性 | ✅ PASS | proto 生成零 diff |
| D8 字段全链路对齐 | ✅ PASS | 关键字段在所有技术栈中均存在 |
| D4 编译零 Warning | ✅ PASS | go vet / vue-tsc / cmake 均无 warning |
| D1 测试覆盖率 | ✅ PASS | 整体 XX% ≥ 60% |

## 遗留问题
- 无 / 列出未修复项及原因
```

### 4.4 输出验收报告

```markdown
# 设计对齐验收报告

## 验收结论
{通过 / 有条件通过 / 阻塞}

## 逐维度详情

| 维度 | 判定 | 阈值 | 实际 | 证据 |
|------|------|------|------|------|
| D8 契约一致性 | PASS | zero diff | 0 diff | protoc-gen.sh 输出 |
| D8 字段对齐 | PASS | 全技术栈覆盖 | X/Y 文件含目标字段 | grep 结果 |
| D4 编译 | PASS | 0 warning | 0 warning | go vet / vue-tsc 输出 |
| D1 覆盖率 | PASS | ≥60% | XX% | coverage.out |
```

## 关键约束

| 约束 | 说明 |
|------|------|
| 先计划后执行 | 阶段一产出修复计划之前，不可修改任何代码文件 |
| 用户确认关 | 阶段二用户未确认前，不可进入阶段三 |
| 只增不删原则 | proto3 新增字段优先；旧字段标记 deprecated 注释但不删除（避免破坏已有测试） |
| contracts 先行 | proto/openapi 变更永远在第一并行组 |
| 失败即停 | 子代理失败立即暂停，不自行修复，报告用户等待指示 |
| 只改目标文件 | 子代理只修改任务清单中列出的文件，不顺手改其他文件 |
| 验收必须实际运行 | 验收标准必须运行工具并读取实际输出，不可推测结果 |

## 与其他 Skill 的关系

```
design-drift-audit (上游)
    │  产出: 漂移审计报告
    ↓
design-alignment-remediation (本 Skill)
    │  消费审计报告 → 修复计划 → 执行 → 验收报告
    ↓
acceptance-gate (下游, 可选)
    │  消费验收报告 → 门禁评审
    ↓
progress-recorder (全程伴随)
```
