# Design Drift Audit（设计漂移审计）

## 定位

实施阶段的**周期性检查点**。职责边界明确：

- **做**: 核对"设计文档声称了什么"与"代码实际有什么"，发现差距
- **不做**: 任何代码修改。修复是下游 `design-alignment-remediation` 的职责

漂移审计的价值在于**越早发现越便宜**。如果在下一轮编码开始前发现数据结构定义不一致，修复成本是改几个字段；如果等到联调阶段才发现，修复成本是重写整个数据链路。

## 唯一事实来源

**设计文档是唯一事实来源。代码的正确性以设计文档为准。**

两个铁律：

1. **设计有、代码没有 → 结构性缺失，按严重级别标记**
2. **代码有、设计没有 → 冗余/未登记，标记为需要确认意图**

绝不反向推导。不因为"代码已经这么写了"而推断"设计大概就是这个意思"。不因为"代码跑通了"而认为"不需要对齐设计"。

### 事实来源定位策略

按以下优先级确定设计文档位置：

**优先级 1 — 用户显式指定**：用户在调用时直接提供设计文档路径或目录。

**优先级 2 — 项目约定**：按以下顺序查找项目中的设计产出目录：
1. `artifacts/` 下的阶段子目录（如 `detailed-design/`、`architecture/`）
2. `docs/design/`
3. `specs/`
4. 项目根目录下包含 `design`、`spec`、`architecture` 关键词的目录

**优先级 3 — 交互确认**：若前两步均未找到，向用户提问"请提供设计文档的路径"并将答案写入本次会话上下文。

### 版本选择

在找到的设计文档目录中，始终选择**最新批准版本**。判断方式：
- 文件名中的日期戳（取最新）
- 版本号标记（取最高）
- 交接摘要中的 `status: ready` 或 `status: approved`

排除 `draft`、`WIP`、`review` 状态的文档。

## 审计标准

### 审计覆盖范围

**"每一行设计 vs 每一行代码"** — 不抽样、不跳跃。设计文档中每一个数据对象、接口、组件、字段定义都必须找到对应的代码实现（或标记为缺失）。

代码侧的检查范围包括但不限于：
- 接口契约文件（proto、openapi、graphql schema 等）
- 数据库定义文件（DDL、migration、ORM 实体等）
- 后端业务代码（struct、model、handler、serializer 等）
- 前端类型定义（TypeScript interface、type、enum 等）
- 算法/核心逻辑代码（class、struct、函数签名等）

### 三个审计维度

```
维度1: 存在性
  问题: 设计定义了 X，代码中 X 是否存在？
  检查: 遍历设计基线每一项，在代码中搜索对应定义
  结果: 存在 / 部分存在 / 完全缺失

维度2: 完整性
  问题: X 在代码中存在，但设计的每个字段/属性是否齐全？类型是否一致？
  检查: 对"存在"的项，逐字段比对名称、类型、约束
  结果: 完全一致 / 字段缺失 / 类型不一致 / 命名不一致

维度3: 跨栈一致性
  问题: 同一个概念在各技术栈（契约层/数据库层/后端/前端/算法）的定义是否一致？
  检查: 选取一个概念（如"轨迹点"），在 proto/DDL/Go struct/TS type/C++ struct 间对比
  结果: 全栈一致 / 部分栈不一致（列出不一致的技术栈对）
```

### 反方向检查

审计不只看"设计→代码"，也要做"代码→设计"的反向扫描：

- 代码中存在但设计文档中无记载的数据结构、接口、组件 → 标记为**未登记项**
- 未登记项不一定是错误（可能是实现细节、内部工具类），但需要**显式确认**：
  - 确认是设计遗漏 → 补充设计文档
  - 确认是实现细节 → 标记为 `by-design`，不纳入漂移计数

### 严重级别定义

| 级别 | 判定标准 | 典型场景 |
|------|----------|----------|
| **CRITICAL** | 结构性缺失。整个接口/表/组件/消息类型在设计中有定义，代码中完全不存在。阻塞依赖此项的下游工作。 | 设计定义了 `CameraConfig` 表和相关 CRUD，代码完全没有 |
| **MAJOR** | 字段级缺失或类型不一致。代码存在但定义与设计不符，会导致集成时数据错误。 | 设计定义 `pile_number` 为 int64，代码仍使用旧的 `x,y,z` float |
| **MINOR** | 冗余或命名偏差。影响可读性和维护，但不影响当前功能。 | 代码保留了设计已废弃但标记为 deprecated 的字段 |

**级别决定影响评估**：CRITICAL 阻塞下一轮编码，MAJOR 在联调时会出错，MINOR 可以延后处理。

## 执行流程

### Step 1: 定位事实来源

```
动作:
  1. 按"事实来源定位策略"(优先级1→2→3)确定设计文档位置
  2. 确认版本（取最新已批准版本）
  3. 列出所有纳入审计的设计文档清单
  4. 确定代码检查范围（根据项目目录结构识别所有技术栈的代码目录）

输出:
  - 设计文档清单（路径 + 版本）
  - 代码范围清单（目录 + 技术栈）
  - 若用户指定了路径 → 优先使用用户指定
  - 若走约定发现 → 向用户确认"我找到了以下设计文档作为审计基线：[清单]，是否正确？"
```

### Step 2: 提取设计基线

```
动作:
  遍历每份设计文档，提取三类"声称"为结构化清单：

  A. 数据对象
     - 对象名称（表名、struct名、type名）
     - 字段列表：字段名、数据类型、是否必填、默认值、约束
     - 来源文档 + 行号引用

  B. 接口契约
     - 接口名称（RPC方法、REST端点、消息类型）
     - 输入结构：字段名 + 类型
     - 输出结构：字段名 + 类型
     - 协议类型（gRPC/HTTP/WS/IPC）
     - 来源文档 + 行号引用

  C. 组件定义
     - 组件名称
     - 对外接口列表
     - 依赖的其他组件
     - 来源文档 + 行号引用

输出: 设计基线清单（结构化，每一项可独立核对）
```

提取时注意：
- 设计文档中的引用关系需要展开（如 "详见 G303 xxx 文档" → 实际读取该文档）
- 设计文档中定义的类型映射关系需要记录（如 "int64 在 proto 中对应 int64，在 Go 中对应 int64，在 TS 中对应 number"）
- 如果设计文档本身有歧义（同一概念两处定义不一致），先标记为"设计待澄清"，不纳入漂移计数

### Step 3: 发现代码现实

```
动作:
  对设计基线的每一项，在代码中搜索对应实现：

  搜索策略（按技术栈适配）:
  - 接口契约层(proto/openapi): 直接读取接口定义文件，提取 message/endpoint/schema
  - 数据库层(DDL/ORM): 读取 migration 文件或 ORM 实体定义
  - 后端代码: grep struct/class/interface 定义，读取 handler/controller
  - 前端代码: grep type/interface/enum 定义
  - 算法代码: grep struct/class/函数签名 定义

  搜索原则:
  - 优先用名称匹配（设计中的"TrajectoryPoint"→搜索代码中的"TrajectoryPoint"）
  - 名称匹配不到时，尝试语义匹配（搜索相关关键词）
  - 两次匹配都失败 → 标记为"缺失"
  - 匹配成功 → 提取代码中的实际定义，进入字段级比对

输出: 代码现实清单（与设计基线条目一一对应的结构）
```

关键行为约束：
- **不要被代码中的注释误导**：代码注释写的"TODO: align with v2.1"不等于已对齐
- **不要被测试通过误导**：测试覆盖的是旧结构，不代表新结构已实现
- **生成代码（如 protoc 生成的 .pb.go）不作为检查目标**：检查源文件（.proto），生成代码的一致性由 CI 保证

### Step 4: 逐项核对

```
动作:
  将设计基线清单与代码现实清单逐项对比，执行三个维度的检查：

  A. 存在性检查
     对每一项设计基线:
       → 代码中有对应项 → PASS
       → 代码中无对应项 → 标记为 DRIFT-{NN}，维度=存在性
       → 代码中有部分（如只有 struct 定义没有 handler）→ 标记为 DRIFT-{NN}，维度=存在性，备注"部分缺失"

  B. 完整性检查
     对每一项"存在"的项:
       → 逐字段比对名称 → 不一致标记
       → 逐字段比对类型 → 不一致标记（注意类型映射：设计中的"int64"在 Go 中是"int64"，在 TS 中是"number"）
       → 逐字段比对约束 → 不一致标记（如设计要求 NOT NULL，代码中是可空指针）
       → 全部一致 → PASS

  C. 跨栈一致性检查
     选取设计中的核心数据对象（在所有技术栈中都应有对应定义的概念）:
       → 提取各栈的实际定义
       → 横向对比字段是否一致
       → 任一栈不一致 → 标记为 DRIFT-{NN}，维度=跨栈一致性

  D. 反方向检查（代码→设计）
     遍历代码中的重要定义（struct/interface/message/table）:
       → 在设计基线中搜索对应定义
       → 找不到 → 标记为未登记项，询问用户意图
```

**类型映射表**（核对完整性时参考，以设计文档中的声明为准）：

| 设计中的抽象类型 | Proto | SQL | Go | TypeScript | C++ |
|-----------|-------|-----|-----|------------|-----|
| integer(64) | int64 | BIGINT | int64 | number | int64_t |
| integer(32) | int32 | INTEGER | int32 | number | int32_t |
| float(32) | float | REAL | float32 | number | float |
| float(64) | double | DOUBLE PRECISION | float64 | number | double |
| string | string | VARCHAR/TEXT | string | string | std::string |
| boolean | bool | BOOLEAN | bool | boolean | bool |
| timestamp | - | TIMESTAMPTZ | time.Time | string(ISO) | uint64_t(ms) |
| bytes | bytes | BYTEA | []byte | Uint8Array | std::vector\<uint8_t\> |
| array\<T\> | repeated T | ARRAY | []T | T[] | std::vector\<T\> |

### Step 5: 分类定级

```
动作:
  对 Step 4 产生的每个差异项:

  1. 按严重级别标准分类:
     - 整个对象/接口/表缺失 → CRITICAL
     - 字段缺失/类型不一致 → MAJOR
     - 冗余/命名偏差 → MINOR

  2. 评估下游影响:
     - 列出哪些未完成的计划/任务依赖此项
     - 判断是否会阻塞下一轮编码的启动

  3. 检查是否已有对应的技术债务登记:
     - 若 tech-debt-tracker 中已有此缺口 → 引用 TD-ID
     - 若没有 → 建议登记

输出: 带严重级别、影响评估、TD引用的漂移项清单
```

### Step 6: 生成审计报告

生成两种格式的输出：

**格式 1 — Markdown 报告**（给人看，保存到 `docs/` 目录）

**格式 2 — JSON 数据**（给下游 Skill 消费，保存到审计报告同目录）

具体的输出模板和 JSON Schema，见 `references/output-templates.md`。

报告保存路径：`docs/superpowers/specs/YYYY-MM-DD-design-drift-audit-report.md` 和同目录下的 `.json` 文件。

## 与其他 Skill 的关系

```
implementation-plan-generation
    ↓ 产出可执行计划
[计划执行，持续产出代码]
    ↓
design-drift-audit  ← 本 Skill
    │ 触发时机：每轮编码完成后 / 设计更新后 / 下轮开始前 / 用户要求
    ↓ 产出：审计报告(MD + JSON)
design-alignment-remediation  ← 下游消费者
    │ 消费审计报告中的漂移项清单，生成修复计划
    ↓ 产出：对齐后的代码
acceptance-gate  ← 验证
    │ 验证修复是否通过八维度验收
```

本 Skill 与 `plan-completion-gate-review` 的分工：
- `plan-completion-gate-review`: 检查**过程合规性**（任务是否假完成、状态是否跳级）
- `design-drift-audit`: 检查**内容一致性**（代码和设计是否一致）

两者互补，互不替代。

## 关键行为原则

### 原则 1: 设计驱动，而非代码驱动

审计的唯一标尺是设计文档。如果代码已经"跑通了"但与设计不一致，代码就是错的。如果设计有歧义，标记为"设计待澄清"而非"以代码为准"。

原因：代码是一个时间点的快照，设计是经过评审的意图。用代码反推设计等于放弃设计评审的价值。

### 原则 2: 全面覆盖，不做抽样

设计文档中的每一项声称都必须被检查。不对"看起来不重要"的字段跳过检查——一个小小的类型不一致在联调时会变成难以定位的 bug。

原因：漂移的伤害不是线性的。一个字段的类型漂移可能导致整个数据链路的静默错误。

### 原则 3: 只审计，不修复

审计报告是"体检报告"，不是"手术方案"。发现 CRITICAL 问题时标记它、评估影响、输出报告，但不直接动手修改代码。

原因：修复需要理解设计意图、评估变更范围、制定修复策略——这些是 `design-alignment-remediation` 的职责。混在一起会导致审计不彻底（"反正要改，就不记了"）或修复不系统（"看到什么改什么"）。

### 原则 4: 输出即合约

审计报告是下游 Skill（`design-alignment-remediation`）的输入合约。JSON 输出中的每个字段都是下游生成修复任务的数据来源。格式不完整或数据缺失会导致下游无法自动化消费。

原因：漂移审计 → 漂移修复 这条链路的价值在于自动化。如果下游需要人工重新解读报告，就失去了结构化输出的意义。

### 原则 5: 环境感知，不自说自话

不预设项目使用什么技术栈、设计文档放在哪里、代码目录怎么组织。每一步都从项目实际状态出发：先扫描、再确认、后执行。

原因：这个 Skill 要服务于不同类型的项目（monorepo/多仓库、不同语言组合、不同设计文档组织方式）。硬编码路径或技术栈等于限制它的适用范围。
