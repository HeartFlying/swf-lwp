# Feature 详细设计 Skill

## 目标

将系统级详细设计转化为单 Feature 的可研发制品。核心定位是**上游设计的消费者和具体化者**，而非从零开始设计。

- **输入侧**：从上游设计制品中提取该 Feature 相关的设计决策
- **输出侧**：将提取的设计决策转化为研发可直接使用的 DDL、接口定义、代码骨架、任务清单
- **垂直切片**：一个 Feature 贯通数据层 → 服务层 → 接口层 → 前端层

## 上游依赖与输入消费

本 Skill 消费上游详细设计阶段的制品。执行前，必须能访问以下类型的设计文档（实际路径从项目 CLAUDE.md 或 roadmap 文档中定位）：

### 1. 范围基线文档 (P0 — 必选)

**必须包含的内容**：
- 功能标识到组件的映射表（每行含功能 ID、功能描述、优先级、覆盖的组件列表）
- 功能是否有前端界面的标记
- 功能优先级（Must/Should/Could）

**消费方式**：从中提取目标 Feature 的一行映射记录，作为该 Feature 的权威范围基线。

### 2. 组件设计文档 (P0 — 必选)

**必须包含的内容**：
- 组件目录（组件 ID、名称、职责描述）
- 组件的内部结构（模块划分、依赖方向）
- 关键状态机或状态迁移定义
- 核心算法/规则的描述
- 异常处理策略与恢复边界
- 面向接口和数据设计的交接边界字段

**消费方式**：提取该 Feature 涉及的所有组件的职责、状态机、算法、异常处理定义。

### 3. 接口设计文档 (P1 — 条件必选)

**必须包含的内容**：
- 接口目录（接口 ID、名称、提供方、消费方、交互类型）
- 请求/响应契约字段
- 错误码分层定义（含调用方责任和降级建议）
- 幂等与重试语义
- 版本与兼容策略

**消费方式**：提取该 Feature 涉及的接口契约。**条件**：仅当上游执行了接口设计且该 Feature 涉及对外接口时必须。

### 4. 数据设计文档 (P1 — 条件必选)

**必须包含的内容**：
- 数据对象目录（对象 ID、名称、属性族、关系）
- 数据所有权与读写边界定义
- 存储边界和访问层次
- 一致性级别与并发约束
- 数据版本演化与迁移策略

**消费方式**：提取该 Feature 涉及的数据对象、所有权、存储边界、一致性要求。**条件**：仅当上游执行了数据设计且该 Feature 涉及数据存储时必须。

### 5. 技术栈与就绪度文档 (P2 — 按需)

**可能包含的内容**：技术栈选型、项目结构约定、工程开工准备度评估。

**消费方式**：用于确定代码骨架的语言/框架、产出路径结构。

### 输入缺失处理

- 范围基线缺失 → 阻塞，必须由上游补全功能映射
- 组件设计缺失 → 阻塞，无法确定 Feature 涉及的组件边界
- 接口设计缺失但有对外接口 → 降级为自主定义接口契约，记录为工作假设
- 数据设计缺失但有数据存储 → 降级为自主设计表结构，记录为工作假设
- 技术栈文档缺失 → 从项目代码中自动检测技术栈

## 输出规范

为每个 Feature 创建独立目录，包含以下标准文件：

```
docs/features/F{n}-{feature-name}/
├── 00-traceability-matrix.md     # 上游回链追溯矩阵
├── 01-ddl.sql                    # 数据库DDL脚本
├── 02-openapi.yaml               # OpenAPI 3.0 接口定义（含接口规范说明）
├── 03-code-skeleton/             # 代码骨架目录
│   ├── backend/                  # 后端代码骨架
│   ├── frontend/                 # 前端代码骨架（条件产出）
│   └── algorithm/                # 算法侧代码骨架（条件产出）
├── 04-task-breakdown.md          # 垂直切片研发任务拆解
└── 05-acceptance-tests.md        # 验收测试用例
```

## 执行步骤

### Step 1: Feature 范围提取与上游回链

**核心任务**：从上游设计制品中精确提取该 Feature 的设计决策，建立追溯链。不重新分析需求，而是消费已有的设计结论。

**执行动作**：

1. **从范围基线中定位该 Feature**
   - 在功能-组件映射表中找到该 Feature 对应的行
   - 提取：功能 ID、功能描述、优先级、覆盖的组件列表、是否有前端界面
   - 如找不到，要求用户明确 Feature 名称或功能 ID

2. **从组件设计中提取 Feature 涉及的组件**
   - 对范围基线中列出的每个组件 ID，在组件设计文档中提取：
     - 组件职责描述
     - 内部模块结构与依赖方向
     - 关键状态机定义（状态对象、迁移条件、触发事件）
     - 核心算法/规则（输入输出、复杂度约束、失败语义）
     - 异常处理策略（检测点、恢复/补偿/降级方案）
     - 面向接口和数据设计的交接边界字段

3. **从接口设计中提取 Feature 涉及的接口**（如有）
   - 对每个相关接口，提取：
     - 接口契约（请求/响应字段、类型、约束）
     - 错误语义（错误码分层、调用方责任）
     - 幂等与重试策略
     - 版本与兼容约束

4. **从数据设计中提取 Feature 涉及的数据对象**（如有）
   - 对每个相关数据对象，提取：
     - 属性族与关系
     - 所有权归属和读写边界
     - 一致性级别要求
     - 存储边界约束
     - 演化与迁移注意事项

5. **建立追溯矩阵** → 产出 `00-traceability-matrix.md`

   模板结构：
   ```
   # {Feature} 上游追溯矩阵

   ## 功能范围
   | 功能ID | 功能描述 | 优先级 | 来源文档 |

   ## 组件追溯
   | 组件ID | 职责摘要 | 涉及的状态机 | 涉及的算法 | 来源章节 |

   ## 接口追溯
   | 接口ID | 契约摘要 | 错误语义 | 幂等策略 | 来源章节 |

   ## 数据追溯
   | 数据对象ID | 所有权 | 一致性级别 | 存储边界 | 来源章节 |

   ## 未覆盖项
   | 设计关注点 | 缺失原因 | 工作假设 | 风险等级 |
   ```

**完成标志**：追溯矩阵中每个设计决策都有明确的上游来源引用。

---

### Step 2: 数据层具体化

**核心任务**：将上游数据设计决策转译为可执行的 DDL 脚本。不是从零设计表结构，而是将数据对象/边界/一致性决策落地为具体 SQL。

**设计原则**：
- 表结构必须反映上游数据设计中定义的所有权和存储边界
- 索引设计要覆盖上游定义的读写模式
- 约束条件要体现上游定义的一致性级别
- 字段类型选择要参考项目实际使用的数据库

**执行动作**：

1. 从追溯矩阵中提取该 Feature 涉及的全部数据对象
2. 为每个数据对象设计对应的物理表结构
3. 根据读写边界定义索引策略（读多写少 → 多索引，写多读少 → 少索引）
4. 根据一致性级别添加约束（外键、唯一、CHECK）
5. 参考 `references/ddl-template.sql` 的标准结构（主键、时间戳、触发器、注释）

**产出物**：`01-ddl.sql`

---

### Step 3: 接口层具体化

**核心任务**：将上游接口设计契约转译为可交付的 OpenAPI 定义。如有上游接口设计，严格遵循其契约字段；如无，则基于组件交接边界自主定义。

**设计原则**：
- 接口路径和请求/响应字段优先从上游接口设计提取
- 错误响应码必须对齐上游定义的错误语义分层
- 幂等键和重试条件必须反映上游设计的幂等策略
- 接口描述中嵌入调用方责任和降级建议（替代独立的接口规范文档）

**执行动作**：

1. 从追溯矩阵中提取该 Feature 涉及的全部接口契约
2. 按 OpenAPI 3.0 规范定义路径、参数、请求体、响应体
3. 为每个端点定义完整的错误响应（对齐上游错误语义）
4. 在 `description` 字段中补充：
   - 调用方责任
   - 超时/重试策略
   - 降级建议
5. 参考 `references/openapi-template.yaml` 的格式规范

**产出物**：`02-openapi.yaml`

---

### Step 4: 业务逻辑骨架

**核心任务**：将上游组件设计中的职责、状态机、算法/规则转化为代码骨架。代码骨架不是空壳，应反映上游设计的结构。

**技术栈检测**：先扫描项目根目录和主要代码目录（如 `go.mod`、`CMakeLists.txt`、`package.json`），确定实际使用的语言和框架，而非使用预设模板。

**执行动作**：

1. 确定 Feature 归属的服务模块（边缘侧/中心侧/算法侧）
2. 根据项目实际技术栈生成代码骨架：
   - **Go 后端**：生成 service 层接口定义、handler 骨架、model 结构体
   - **C++ 算法**：生成类声明、关键方法签名、CMakeLists 片段
   - **其他语言**：参考 `references/code-skeleton-examples.md` 的对应章节
3. 在骨架中标注：
   - 状态机实现锚点（引用上游状态迁移定义）
   - 算法实现锚点（引用上游算法描述和复杂度约束）
   - 异常处理锚点（引用上游失败模式和恢复路径）
4. 骨架代码应能通过编译/语法检查

**产出物**：`03-code-skeleton/backend/`、`03-code-skeleton/algorithm/`

---

### Step 5: 前端骨架（条件执行）

**触发条件**：仅在 Step 1 中确认该 Feature 有前端界面需求时执行。

**执行动作**：

1. 从接口设计中提取前端需要消费的接口契约
2. 生成 TypeScript 类型定义（对齐接口响应字段）
3. 生成 API 客户端封装
4. 生成页面组件骨架（列表页/表单页/详情页，按 Feature 特性选择）
5. 前端技术栈（Vue3/React/Angular）从项目现有前端目录检测

**产出物**：`03-code-skeleton/frontend/`

---

### Step 6: 垂直切片任务拆解

**核心任务**：按垂直切片方式组织研发任务，从数据层→服务层→接口层→前端层依次排列。

**执行动作**：

1. 将 Feature 实现拆解为垂直顺序的任务序列：
   - Phase 1 — 数据就绪：DDL 执行、数据访问层
   - Phase 2 — 业务逻辑：状态机实现、算法实现、服务层
   - Phase 3 — 接口暴露：API 实现、契约测试
   - Phase 4 — 前端消费（如适用）：页面实现、联调
   - Phase 5 — 验收：测试用例执行、质量门验证
2. 每个任务标注：预估工时、上游依赖任务、对应的设计制品引用
3. 标注关键路径和可并行任务

**产出物**：`04-task-breakdown.md`

---

### Step 7: 验收测试用例

**核心任务**：定义 Feature 的验收测试用例，覆盖正常路径、异常路径和边界条件。

**设计原则**：
- 测试场景应回链到上游组件设计中的状态机和异常处理定义
- 测试数据应反映上游数据设计中的边界条件
- 如有上游质量门结论，优先覆盖质量门中标记的风险点

**执行动作**：

1. 从组件设计的异常处理定义中提取异常测试场景
2. 从状态机定义中提取状态迁移测试场景
3. 定义单元测试、集成测试、端到端测试用例
4. 参考 `references/acceptance-tests-template.md` 的格式

**产出物**：`05-acceptance-tests.md`

## 模式感知

上游详细设计阶段可能以不同深度执行。根据可用的上游制品，自动调整本 Skill 的执行范围：

| 可用上游制品 | 执行范围 | 降级处理 |
|-------------|---------|---------|
| 范围基线 + 组件设计 + 接口设计 + 数据设计 | **完整执行** Step 1-7 | 无降级 |
| 范围基线 + 组件设计 + 接口设计（无数据设计） | Step 1-7，数据层降级 | Step 2 基于组件交接边界自主设计表结构，记录为工作假设 |
| 范围基线 + 组件设计（无接口设计、无数据设计） | Step 1-7，接口层和数据层降级 | Step 2-3 基于组件交接边界自主设计，记录工作假设 |

**降级原则**：
- 降级不是失败，而是用工作假设填补缺失上游制品的空白
- 每个工作假设必须显式记录在追溯矩阵的"未覆盖项"章节
- 降级产出的设计决策在后续评审中应优先校验

## 技术栈检测与适配

本 Skill 不预设技术栈。执行时按以下顺序确定：

1. **读取项目 CLAUDE.md** → 获取项目级技术栈声明
2. **扫描项目文件** → `go.mod` → Go；`CMakeLists.txt` → C++；`package.json` → Node/前端
3. **检查现有代码目录结构** → 确定模块划分和命名约定
4. **如无法自动检测** → 询问用户确认技术栈

代码骨架产出必须对齐项目现有技术栈和目录约定，而非使用预设模板语言。

## 质量检查清单

交付前自检以下项目：

- [ ] Step 1 追溯矩阵中每个设计决策都有上游来源引用
- [ ] DDL 脚本可直接执行，表结构反映上游数据设计的所有权和存储边界
- [ ] OpenAPI 符合 3.0 规范，错误响应码对齐上游错误语义分层
- [ ] 接口定义包含调用方责任、超时/重试策略和降级建议
- [ ] 代码骨架使用项目实际技术栈（非预设），能通过编译/语法检查
- [ ] 代码骨架中标注了状态机锚点、算法锚点、异常处理锚点
- [ ] 任务拆解按垂直切片顺序排列，每个任务有上游设计引用
- [ ] 测试用例覆盖上游定义的异常场景和状态迁移路径
- [ ] 降级项已显式记录工作假设
- [ ] 所有文档使用项目约定语言编写

## 分阶段执行策略

对于复杂 Feature，建议分阶段执行：

| 阶段 | 步骤 | 内容 | 产出 |
|------|------|------|------|
| **阶段A** — 提取与对齐 | Step 1 | 范围提取 + 上游回链 + 追溯矩阵 | `00-traceability-matrix.md` |
| **阶段B** — 数据与接口 | Step 2-3 | DDL + OpenAPI | `01-ddl.sql`、`02-openapi.yaml` |
| **阶段C** — 骨架与任务 | Step 4-6 | 代码骨架 + 任务拆解 | `03-code-skeleton/`、`04-task-breakdown.md` |
| **阶段D** — 验收 | Step 7 | 测试用例 | `05-acceptance-tests.md` |

每阶段完成后立即保存文件。复杂 Feature 判断标准：涉及 5 个以上组件 或 3 个以上数据对象 或 有前端界面。

## 注意事项

1. **消费者定位**：本 Skill 消费上游设计决策，不重新做设计。如果发现上游设计缺失关键信息，记录为工作假设并标注风险，而非替代上游做设计决策
2. **具体化而非翻译**：DDL 和代码骨架是上游抽象设计的具体化，需补充实现细节但不应偏离上游设计意图
3. **回链优先**：每个实现决策都要能追溯到上游设计来源。如果某决策无法回链，说明它可能是过度设计或范围蔓延
4. **可执行性**：DDL 和代码骨架必须是可直接使用的，而非伪代码
5. **垂直完整性**：一个 Feature 的产出物应覆盖该 Feature 涉及的完整技术栈层次，不留下跨层空白
