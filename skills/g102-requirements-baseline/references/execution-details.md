# G102 执行细则

## 1. 目标

补充 `SKILL.md` 的执行级约束，确保 G102 在不同模式下均能稳定产出可冻结的需求基线。

## 2. 执行前检查

1. 确认 `RA-04` 已进入 `drafting`，并完成台账回写。
2. 读取 `001-requirements-intake.md`，确认目标、范围、约束字段是否可解析。
3. 若存在 `001a-requirements-clarification.md`，先生成“澄清闭环表”再开始需求抽取。
4. 若 `G101` 不存在，记录“背景输入缺失声明”，避免隐式假设污染。

方法目录基线：

- [requirements-methods-catalog.md](../../_shared/requirements-methods-catalog.md)

## 3. 澄清闭环规则

1. `001a` 中每条关键澄清必须在基线文档中有唯一去向：`采纳/拒绝/待定`。
2. 标记 `待定` 时必须提供“解除条件 + 时点 + 责任方”，不得仅写自由文本说明。
3. 不允许跳过未闭环澄清直接输出 `done` 态结论。

## 4. 编号与可追溯规则

1. 功能需求编号：`FR-001` 起连续，不允许复用。
2. 非功能需求编号：`NFR-001` 起连续，不允许复用。
3. 验收标准编号：`AC-001` 起连续，且至少与一个需求绑定。
4. 约束编号：`CST-001` 起连续；风险必须使用 `R-001` 序列，待定事项必须使用 `O-001` 序列。
5. 每条 `Must/Should` 需求必须同时具备：`来源 -> 优先级 -> 验收标准 -> 风险` 链路。
6. 追溯矩阵必须覆盖 `FR`、`NFR`、`CST` 三类对象，不得只覆盖功能需求。

## 5. Fast 模式特例

1. Fast 模式允许无 `G101` 输入。
2. 但仍必须产出完整的 FR/NFR/约束/优先级/验证章节，不允许降级为摘要。
3. 若缺少行业背景导致结论不稳定，必须在“未决事项与风险”标记并给出影响说明。

## 6. 常见失败模式

1. 只列“需求描述”，未给出验收标准。
2. 只做 MoSCoW，未建立依赖和冲突说明。
3. 澄清记录存在，但未形成闭环映射。
4. 约束仅列举，不区分强弱约束或解除条件。
5. 追溯矩阵只覆盖 FR，漏掉 `NFR` 或 `CST`。

## 7. 最小自检清单

提交评审前至少确认：

1. 主文档路径正确且章节完整。
2. FR/NFR/AC 编号唯一且无断号。
3. `Must/Should` 需求 100% 有验收标准与风险链路。
4. 澄清闭环表无“无结论”项。
5. `review_result` 未被误写为质量结论。
