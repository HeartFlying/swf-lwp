# G201 执行细化说明

## 1. 作用

本文件用于细化 `G201` 的执行方式，补齐 `SKILL.md` 中“如何真正落地到技术策略文档”的步骤约束。

## 2. 执行原则

1. `G201` 必须以 `G202` 冻结后的目标、驱动和原则为边界，不得脱离上游愿景单独选型。
2. 每项技术推荐都必须说明推荐理由、替代方案和不采用原因。
3. 输出必须直接满足 `G203` 的最小消费字段，不得只停留在讨论层。
4. 子代理只负责起草文档；运行时任务状态、返工入口、`review_result` 和 `resume_from` 由 `G200` 在验收后统一回写。

## 3. 输入消费顺序

1. 优先读取 `artifacts/architecture/002-architecture-vision.md`。
2. 再读取 `artifacts/architecture/001-architecture-intake.md`。
3. 如需补足业务和范围约束，再读取 requirements 阶段主输出与评审结果。

## 4. 步骤级要求

### 4.1 策略目标

1. 必须明确与 `G202` 对齐的目标、驱动和质量属性。
2. 必须列出不可违反的策略边界。

### 4.2 技术路线候选

1. 至少覆盖应用架构、数据与集成、运行部署三类技术域。
2. 每类至少给出 1 个推荐方案和 3 个备选方案；确无备选时说明原因。

### 4.3 关键选型与权衡

1. 至少输出 3 个关键决策主题。
2. 每个决策都要写清收益、代价和触发条件。
3. 权衡结论至少覆盖性能、安全、成本、交付速度四个维度。
4. 技术栈选型必须以技术策略层为粒度进行收敛，不得进入组件实现级设计。
5. 技术栈选型应覆盖与目标架构相关的核心技术能力领域，例如：

   - 前端技术栈
   - 后端技术栈
   - 数据存储技术栈
   - 消息与集成技术栈
   - 容器与运行时技术栈
   - 云平台与基础设施技术栈
   - 安全与身份认证技术栈
   - 可观测性技术栈

6. 每项技术栈选型至少包含：

   - 选型主题
   - 推荐方案
   - 备选方案
   - 推荐理由
   - 不采用原因
   - 适用条件
   - 风险与限制

7. 技术栈选型结论应服务于架构策略决策，不要求输出具体版本号、部署参数、接口规范或组件设计细节；相关内容由 G203 在蓝图阶段进一步固化。

### 4.4 实施约束与 ADR 候选

1. 必须至少给出 3 条实施约束。
2. 必须至少给出 2 个 ADR 候选。
3. 每个 ADR 候选都要明确为何需要在 `G203` 中继续固化。

## 5. 与 G203 的衔接要求

进入 `G203` 前，`G201` 输出至少要保证：

1. 可直接转成蓝图结构主线。
2. 可直接转成组件、部署和集成设计输入。
3. 可直接约束蓝图和 ADR。
4. 可直接作为 ADR 起草清单。

## 6. 质量与恢复要求
1. 中断恢复时仅以 `artifacts/architecture/000-task-tracker.md` 为主锚点，`template.md` 作为辅助证据。
2. 若质量门或评审门未通过，由 `G200` 统一回写：`AD-04` 保持 `status_code=in_progress`、`skill_stage=rework`，并在 `resume_from` 中写明首个返工任务位和待处理问题。
