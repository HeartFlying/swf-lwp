# G203 执行细化说明

## 1. 作用

本文件用于细化 `G203` 的执行方式，补齐 `SKILL.md` 中“如何真正落地到架构蓝图与 ADR 文档”的步骤约束。

方法来源说明：

1. 本文件中涉及的方法名，应以 `../_shared/architecture-methods-catalog.md` 为唯一标准引用来源。

## 2. 执行原则

1. `G203` 必须以 `G201` 的技术策略为边界，不得脱离上游策略自行重写路线。
2. 蓝图必须同时覆盖结构主线、组件边界、数据架构、接口架构、部署运行边界，不得只停留在单一视图。
3. 输出必须直接满足 `G204` 的最小消费字段，不得只写概念性描述。
4. 子代理只负责起草文档；运行时任务状态、返工入口、`review_result` 和 `resume_from` 由 `G200` 在验收后统一回写。

## 3. 输入消费顺序

1. 优先读取 `artifacts/architecture/001-technical-strategy.md`。
2. 再读取 `artifacts/architecture/002-architecture-vision.md`；该文档为必选输入，缺失时不得启动当前 Skill。
3. 如需补足范围与边界约束，再读取 `artifacts/architecture/001-architecture-intake.md` 与 requirements 阶段主输出。

## 4. 步骤级要求

### 4.1 蓝图目标与范围

1. 必须显式写出蓝图覆盖范围和 deferred 内容。
2. 必须形成 `context / structure / interaction / deployment` 四类视图；模板中的 `view_type` 必须使用这组标准枚举，不得自行缩减或改写。
3. 若范围无法冻结，当前轮不得进入 `G204`。
4. 必须执行蓝图追溯映射，至少建立愿景约束、技术策略、蓝图范围和视图集合之间的显式回链。
5. 必须在 `2.3 视图覆盖声明` 中明确 `S5-A02/S5-A03/S5-A04/S5-A05` 分别由哪些视图和结构化字段承接，且覆盖声明要与第 7 章最小字段一一对应。

### 4.2 组件、数据与接口

1. 必须至少给出 5 个核心组件/模块。
2. 必须至少给出 3 条关键交互链路。
3. 每条链路都要说明正常路径和异常路径。
4. 必须显式列出核心数据对象、权威写入位置、读取/派生位置、一致性/主从边界和生命周期要求，不得只在组件说明中隐含带过。
5. 必须显式列出接口边界、调用方式、契约约束、异常语义和集成模式，并能回指至少一条关键链路或一项依赖约束。
6. 必须检查组件、数据、接口与视图之间的一致性，避免结构视图、交互视图和正文描述互相脱节。

### 4.3 部署拓扑与运行边界

1. 必须至少给出 2 个运行节点或部署单元。
2. 必须说明环境边界、高可用或隔离约束。
3. 部署结论必须能映射到组件或链路。
4. 必须执行一次多视图一致性检查，确认部署拓扑与结构视图、数据/接口约束、交互链路和 ADR 约束之间不冲突。

### 4.4 ADR 固化

1. 必须至少给出 2 条 ADR。
2. 每条 ADR 都要满足最小契约：`context / decision / consequences / source_view_ids / source_component_ids`。
3. 每条 ADR 都要标明状态、关联视图/组件和依据。
4. ADR 与蓝图元素之间必须建立显式索引关系；涉及数据权威边界、接口契约或部署约束的关键决策，不得只留在正文中而不固化。

## 5. 与 G204 的衔接要求

进入 `G204` 前，`G203` 输出至少要保证：

1. 可直接转成评审对象清单。
2. 可直接证明 `S5-A02/S5-A03/S5-A04/S5-A05` 在当前蓝图中的承接位置和字段落点。
3. 可直接检查职责边界和依赖合理性。
4. 可直接检查核心数据对象、相关数据流、存储边界、一致性/主从边界和生命周期是否闭环。
5. 可直接检查接口边界、调用方式、契约约束、异常语义和集成模式。
6. 可直接检查关键链路与异常路径。
7. 可直接检查运行边界和部署约束。
8. 可直接检查关键决策是否已固化。
9. 可直接转成评审关注项，并且必须通过 `target_id/related_flow_id/related_adr_id/related_topology_id` 回指具体蓝图元素。
10. 必须可直接供 `G204` 检查 ADR 的最小结构是否完整。
11. 必须能证明 `blueprint_scope -> architecture_views / coverage_declaration -> component_inventory / data_architecture / interface_architecture / interaction_flows / deployment_topology -> adr_index / adr_contract` 的追溯链完整。
12. 必须能证明结构视图、数据/接口约束、交互视图、部署视图与 ADR 之间已经完成一次一致性检查，并保留可回查证据。

## 6. 质量与恢复要求

1. 中断恢复时仅以 `artifacts/architecture/000-task-tracker.md` 为主锚点，`template.md` 作为辅助证据。
2. 若质量门或评审门未通过，由 `G200` 统一回写：`AD-05` 保持 `status_code=in_progress`、`skill_stage=rework`，并在 `resume_from` 中写明首个返工任务位和待处理问题。
3. `G203` 本地验收仅覆盖文档完整性、结构化最小字段和追溯关系；`AD-07/AD-08` 是否通过由 `G200` 联动共享服务独立判定，不并入 `G203` 自身验收结论。
