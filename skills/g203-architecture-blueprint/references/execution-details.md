# G203 执行细化说明

## 1. 作用

本文件用于细化 `G203` 的执行方式，补齐 `SKILL.md` 中“如何真正落地到架构蓝图与 ADR 文档”的步骤约束。

方法来源说明：

1. 本文件中涉及的方法名，应以 `../_shared/architecture-methods-catalog.md` 为唯一标准引用来源。

## 2. 执行原则

1. `G203` 必须以 `G201` 的技术策略为边界，不得脱离上游策略自行重写路线。
2. 蓝图必须同时覆盖结构主线、组件边界、数据架构、接口架构、部署运行边界，不得只停留在单一视图。
3. 输出必须直接满足 `G204` 的最小消费字段，不得只写概念性描述。
4. 子代理只负责起草文档；运行时任务状态、返工入口、`review_result` 和 `resume_from` 由 `G200` 在验收后统一回写。

## 3. 输入消费顺序

1. 优先读取 `artifacts/architecture/001-technical-strategy.md`。
2. 再读取 `artifacts/architecture/002-architecture-vision.md`；该文档为必选输入，缺失时不得启动当前 Skill。
3. 如需补足范围与边界约束，再读取 `artifacts/architecture/001-architecture-intake.md` 与 requirements 阶段主输出。

## 4. 步骤级要求

### 4.1 蓝图目标与范围

1. 必须显式写出蓝图覆盖范围和 deferred 内容。
2. 必须形成 `context / structure / interaction / deployment` 四类视图；模板中的 `view_type` 必须使用这组标准枚举，不得自行缩减或改写。
3. 若范围无法冻结，当前轮不得进入 `G204`。
4. 必须执行蓝图追溯映射，至少建立愿景约束、技术策略、蓝图范围和视图集合之间的显式回链。
5. 必须在 `2.3 视图覆盖声明` 中明确 `S5-A02/S5-A03/S5-A04/S5-A05` 分别由哪些视图和结构化字段承接，且覆盖声明要与第 7 章最小字段一一对应。
6. **MVP 映射要求**：
   - 必须读取 `artifacts/requirements/004-mvp-definition.md`，提取全部 in-scope MVP（Must + Should）。
   - 将 MVP 列表与蓝图 1.2 节的 SCP 列表建立显式映射。若 SCP 数量与 MVP 数量不一致，优先扩展 SCP 项（新增 SCP-XXX）以匹配 MVP，确保下游 `G300` 的 V-01 检查可直接通过。
   - 映射关系必须落入蓝图 3.1 节，字段包括：SCP-ID、MVP-ID、FR-ID、功能描述、优先级、组件覆盖、has_frontend_ui、uiux_ref。

- **系统上下文图要求**：必须在蓝图 2.4 节（或对应位置）生成 Mermaid 系统上下文图，展示 系统边界、外部系统及数据交换方向。图中节点必须使用与正文一致的 scope_id 或 component_id，外部系统边界与 VW-001 定义一致。

### 4.2 组件、数据与接口

1. 必须至少给出 5 个核心组件/模块。
2. 必须至少给出 3 条关键交互链路。
3. 每条链路都要说明正常路径和异常路径。
4. 必须显式列出核心数据对象、权威写入位置、读取/派生位置、一致性/主从边界和生命周期要求，不得只在组件说明中隐含带过。
5. 必须显式列出接口边界、调用方式、契约约束、异常语义和集成模式，并能回指至少一条关键链路或一项依赖约束。
6. 必须检查组件、数据、接口与视图之间的一致性，避免结构视图、交互视图和正文描述互相脱节。

- **组件结构图要求**：必须在蓝图 3.5 节生成 Mermaid 组件分层图，区分边缘侧/中心侧/基础设施/外部系统四层，层间连线标注接口 ID（IFC-XXX）。图中节点必须与 3.2 节 component_inventory 表格一一对应。
- **数据流图要求**：必须在蓝图 3.6 节生成 Mermaid 数据流全景图，覆盖全部核心数据对象（DAT-xxx），箭头标注 data_flow_id。图中数据对象 ID 必须与 3.3 节 data_architecture 表格、3.4 节 data_flows 表格一致。
- **交互时序图要求**：必须在蓝图 4.4 节为所有核心链路生成 Mermaid 时序图，时序图中必须体现异常路径的触发条件。participant 必须与 3.2 节组件清单中的 component_id 一致。

1. **组件清单必须包含 `frontend_consumer` 标记**（yes / no）。凡直接参与前端界面渲染、提供 Web API 供前端调用、或输出前端可视数据的组件，必须标记为 `yes`。
2. **MVP→组件覆盖映射必须完整**：蓝图 3.1 节中，每个 in-scope MVP 的 `组件覆盖` 列不得为空，且所列 CMP-XXX 必须在 3.2 节组件清单中有定义。
3. **前端一致性**：对 `has_frontend_ui=yes` 的 MVP，其 `组件覆盖` 中必须至少包含一个 `frontend_consumer=yes` 的组件；若无法满足，必须在 3.1 节备注中说明原因并标注风险。

### 4.3 部署拓扑与运行边界

1. 必须至少给出 2 个运行节点或部署单元。
2. 必须说明环境边界、高可用或隔离约束。
3. 部署结论必须能映射到组件或链路。
4. 必须执行一次多视图一致性检查，确认部署拓扑与结构视图、数据/接口约束、交互链路和 ADR 约束之间不冲突。

- **部署拓扑图要求**：必须在蓝图 5.2 节生成 Mermaid 部署拓扑图，标注 6 个拓扑节点（TOP-001~006）及其所属 security_zone，节点间通信链路标注对应的 security_protocol（TLS 1.2+ / mTLS / none），外部系统边界（摄像头/卡口/集团平台）与 VW-001 系统上下文图一致。

### 4.4 ADR 固化

1. 必须至少给出 2 条 ADR。
2. 每条 ADR 都要满足最小契约：`context / decision / consequences / source_view_ids / source_component_ids`。
3. 每条 ADR 都要标明状态、关联视图/组件和依据。
4. ADR 与蓝图元素之间必须建立显式索引关系；涉及数据权威边界、接口契约或部署约束的关键决策，不得只留在正文中而不固化。

- **ADR 中Mermaid图要求**：涉及**关键决策并且与蓝图不重复**必须在 ADR 文档中附带 Mermaid 图。图中的状态名称、转换条件必须与 ADR 正文中 decision/consequences 描述一致。

### 4.5 图表一致性检查

在生成架构蓝图文档前，必须执行以下图表校验：

1. **标识符一致性**：图表中出现的所有 CMP-*/DAT-*/IFC-*/TOP-*/FLW-* 标识符，必须在正文对应表格中有定义；反之，正文核心项应在图表中有体现。
2. **关系一致性**：图表中的连线关系（如组件 A → 组件 B）必须与 4.3 节依赖清单（DEP-XXX）或 4.2 节接口清单（IFC-XXX）一致。
3. **安全域一致性**：部署拓扑图中的 security_zone 标注必须与第 5 章表格一致。
4. **渲染可用性**：所有 Mermaid 代码块必须语法正确，可被 GitHub/GitLab 标准 Markdown 渲染器渲染。
5. **互补性检查**：图表不得替代表格，同一信息应在表格（精确字段）和图表（直观结构）中同时存在，形成双轨表达。
6. **异常路径覆盖**：交互时序图必须体现至少 1 条异常路径（如断点恢复、降级处理）。

**不通过条件**：上述任一项检查失败，视为当前轮不可交付，需返回步骤 2~4 修正。

## 5. 与 G204 的衔接要求

进入 `G204` 前，`G203` 输出至少要保证：

1. 可直接转成评审对象清单。
2. 可直接证明 `S5-A02/S5-A03/S5-A04/S5-A05` 在当前蓝图中的承接位置和字段落点。
3. 可直接检查职责边界和依赖合理性。
4. 可直接检查核心数据对象、相关数据流、存储边界、一致性/主从边界和生命周期是否闭环。
5. 可直接检查接口边界、调用方式、契约约束、异常语义和集成模式。
6. 可直接检查关键链路与异常路径。
7. 可直接检查运行边界和部署约束。
8. 可直接检查关键决策是否已固化。
9. 可直接转成评审关注项，并且必须通过 `target_id/related_flow_id/related_adr_id/related_topology_id` 回指具体蓝图元素。
10. 必须可直接供 `G204` 检查 ADR 的最小结构是否完整。
11. 必须能证明 `blueprint_scope -> architecture_views / coverage_declaration -> component_inventory / data_architecture / interface_architecture / interaction_flows / deployment_topology -> adr_index / adr_contract` 的追溯链完整。
12. 必须能证明结构视图、数据/接口约束、交互视图、部署视图与 ADR 之间已经完成一次一致性检查，并保留可回查证据。

- **图表可直接供 G204 评审使用**：评审人员可通过 Mermaid 图表快速理解系统结构、数据流和部署边界，再深入表格核查精确字段。图表降低了评审认知负荷。
- **G204 评审关注点可直接映射到图表**：评审发现的结构问题、依赖冗余、数据流断点等，应在评审结论中标注对应图表位置（如"VW-002 组件结构图中 CMP-003→CMP-004 的连线与 4.3 节 DEP-009 矛盾"）。

## 6. 质量与恢复要求

1. 中断恢复时仅以 `artifacts/architecture/000-task-tracker.md` 为主锚点，`template.md` 作为辅助证据。
2. 若质量门或评审门未通过，由 `G200` 统一回写：`AD-05` 保持 `status_code=in_progress`、`skill_stage=rework`，并在 `resume_from` 中写明首个返工任务位和待处理问题。
3. `G203` 本地验收仅覆盖文档完整性、结构化最小字段和追溯关系；`AD-07/AD-08` 是否通过由 `G200` 联动共享服务独立判定，不并入 `G203` 自身验收结论。
