# 分层启动策略模板参考

## 核心原则

按"依赖层"分三轮启动，每轮产出"可运行的骨架"。POC/压测与编码并行，仅阻塞依赖它的特定模块，不整体阻塞。

## 分层模板

| 轮次 | 目标 | 典型模块 | 验收标准示例 |
|---|---|---|---|
| **第 1 轮（骨架）** | 搭骨架、定契约 | 数据存储 Schema、接口协议层、前端框架、权限/认证基座、设备管理基座 | 数据库可读写、Swagger/UI 可访问、Mock 数据走通主链路、登录鉴权通过 |
| **第 2 轮（核心链路）** | 业务价值最高的端到端链路跑通 | 数据接入 -> 核心处理 -> 输出缓存 | 输入测试数据能走完核心链路，输出格式正确（不要求全并发，要求逻辑正确） |
| **第 3 轮（补齐联调）** | 外围服务 + 全链路集成 | 流媒体、外部对接、监控告警、批量优化、补传/恢复 | 真实/模拟输入 -> 全链路 -> 前端展示闭环，阻塞问题已清零或已切换回退方案 |

## 决策规则

- 若用户没有明确轮次偏好，默认使用三层模板
- 若模块数 < 8，可压缩为两轮（骨架 + 联调）
- 若存在强依赖循环，必须在分层前打破（通过接口契约先行）

## POC/压测并行安排

- 数据库性能类：第 1 轮并行（用 Schema 真表测）
- 网络/跨节点延迟类：第 2 轮并行（核心链路需要时测）
- 客户端稳定性类：第 3 轮并行（联调阶段测）
