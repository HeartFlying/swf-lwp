# 追溯链注入与验收指南

本文件是 `implementation-plan-generation` 步骤 4（注入追溯链）和追溯链验收的参考文档。仅在执行步骤 4 和验收时按需加载。

---

## 1. 追溯链格式规范

放置在每个任务的标题下方：

```
> **追溯链** | 设计: <设计文档路径> §<章节号> | g402: <细化定义文件路径> | 需求: <需求ID列表>
```

**格式要求**：
- `设计` 指向设计文档的具体章节（如 `G302 §3.2.1`、`003-interface-design.md §4.5`）
- `g402` 指向 `artifacts/implementation/` 下的细化定义文件（如 `data-defs/TrackPayload.md`），可选。当 Task 的具体实现依赖 g402 细化定义时，必须添加 g402 引用
- `需求` 指向需求 ID（如 `FCT-003`、`MVP-004`），多个需求用逗号分隔
- 设计文档路径使用相对于仓库根目录的路径
- 需求 ID 必须能在需求文档中定位到（可通过 grep 验证）

---

## 2. 追溯链完整性规则

- 每个 Task 必须有追溯链（无例外）
- 每个需求 ID 必须在至少一个 Task 的追溯链中出现
- 每个设计章节引用必须在至少一个 Task 的追溯链中出现
- 如果某个 Task 覆盖多个需求或设计章节，全部列出

---

## 3. 追溯链验收标准

计划执行完毕后，追溯链完整性是**强制验收项**，不通过则计划不能标记为"完成"。

### 验收方法

1. **数量校验**：统计所有子计划中的 Task 总数和包含追溯链的 Task 数，两者必须相等
2. **格式校验**：每个追溯链必须匹配正则 `> \*\*追溯链\*\* \| 设计: .+ §.+ \| 需求: .+`
3. **引用校验**：每个设计引用路径必须能定位到实际文件，每个需求 ID 必须能在需求文档中 grep 到
4. **覆盖率校验**：所有需求 ID 必须至少在追溯链中出现一次（无孤儿需求）

---

## 4. 验证命令模板

```bash
# 1. 统计追溯链覆盖率
grep -rc "^> \*\*追溯链\*\*" docs/superpowers/plans/
# 预期：每个子计划的 Task 数 = 追溯链行数

# 2. 检查格式合规
grep -r "^> \*\*追溯链\*\*" docs/superpowers/plans/ | grep -vP "设计: .+ §.+ \| 需求: .+"
# 预期：空（无不匹配格式的追溯链）

# 3. 提取所有引用的需求 ID 并与需求文档对照
grep -roP "需求: \K.+" docs/superpowers/plans/ | tr ',' '\n' | sed 's/^ *//' | sort -u
# 人工或脚本核对：每个 ID 是否存在于 artifacts/requirements/ 中
```
