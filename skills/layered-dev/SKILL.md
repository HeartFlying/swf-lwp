# 端到端编码编排（超级技能）

## 核心目标

将上游全部设计产物转化为可并行执行的编码任务，并执行三层验证确保交付质量。

本技能是**设计→代码的最后一道编排层**，不做设计细化（那是 g402 的职责），只做**分发、执行追踪和验证**：

- **Phase 1**：按主计划依赖图，将各执行组派发为独立子代理并行编码。每个子代理收到自包含的派发包（含 g402 细化定义 + g100 需求追溯），无需与其他子代理通信即可独立工作。
- **Phase 2**：集中验证三层一致性——跨组接口契约双边合规、组件实现与 g402 定义一致、数据对象实现与 g402 定义一致。
- **Phase 3**：需求覆盖率验证——全链追溯矩阵（FR→设计→g402定义→代码）、MVP 完整性检查、孤儿代码检测。

核心原则：**设计细化已在 g402 完成，本技能只负责分发和三层验证。**

## Phase 1: 分组并发（Group Parallel Dispatch）

### 目标

按主计划的依赖图，将各执行组派发为独立子代理并行编码。每个子代理收到自包含的派发包（含 g402 细化定义 + g100 需求追溯），无需与其他子代理通信即可独立工作。

### 前置：读取上游产出

本 Phase 不自行创建任何组件/接口/数据对象定义。所有细化定义从 g402 产出中读取：

| 定义类型 | 来源路径 | 用途 |
|----------|----------|------|
| 数据对象定义 | `artifacts/implementation/data-defs/` | 子代理编码时的字段类型权威来源 |
| 接口定义 | `artifacts/implementation/interface-defs/` | 子代理编码时的接口签名权威来源 |
| 组件定义 | `artifacts/implementation/component-defs/` | 子代理编码时的类/方法签名权威来源 |
| 契约制品 | `artifacts/implementation/contracts/` | 子代理编码时的契约合规目标 |
| 需求基线 | `artifacts/requirements/003-requirements-baseline.md` | FR-ID 列表和功能描述 |
| MVP 定义 | `artifacts/requirements/004-mvp-definition.md` | MVP 优先级和范围 |

**若上述路径为空**：中断，要求先执行对应的上游技能。

### 行为规则

1. **读取分组信息**：从主实施计划中提取：
   - 执行组列表（每组包含哪些组件/模块）
   - 组间依赖关系（哪组依赖哪组的输出）
   - 各组的子计划文件路径

2. **判定并行集和串行链**：
   - 无相互依赖的组 → 并行集（可同时启动）
   - 组 A 依赖组 B 的产物 → 串行链（B 完成后才启动 A）
   - 判定依据是主计划的依赖图，不自行重新分析

3. **建立需求→执行组映射**：从实施计划的追溯链中提取每个执行组覆盖的 FR-ID 列表，建立映射表：

   | 执行组 | 覆盖的 FR-ID | MVP-ID |
   |--------|-------------|--------|
   | Plan-A | FR-001, FR-002 | MVP-001 |
   | Plan-B | FR-003, FR-004, FR-005 | MVP-001, MVP-002 |
   | ... | ... | ... |

   此映射表是 Phase 3 需求覆盖率验证的基准。

4. **为每个执行组准备派发包**。派发包必须自包含。派发包包含：

   | 信息项 | 说明 | 来源 |
   |--------|------|------|
   | 组范围 | 该组负责的组件/模块集合及其职责边界 | 子计划 |
   | 任务序列 | 有序的实现任务列表，每个任务附追溯链 | 子计划 |
   | **需求映射** | 该组覆盖的 FR-ID 列表 + 每个 FR 的一句话功能描述 | g100 产出 |
   | **组件定义** | 该组每个组件的细化定义文件（类名、成员变量、方法签名） | g402 产出 |
   | **接口定义** | 该组提供和消费的接口定义文件（参数签名、调用约定） | g402 产出 |
   | **数据对象定义** | 该组涉及的数据对象定义文件（字段类型、约束） | g402 产出 |
   | 契约绑定 | 该组必须遵从的契约制品集合（区分"提供方契约"和"消费方契约"） | g402 产出 |
   | 设计参考 | 相关的架构设计文档路径及具体章节号 | 追溯链提取 |
   | 验收标准 | 该组完成的量化通过/失败条件 | 子计划 |
   | 执行引擎建议 | 推荐的验证模式 | 子计划类型标注 |
   | 前置产出（仅串行组） | 若该组依赖其他组的输出，前置组的已验证产出位置 | Phase 1 追踪 |

5. **派发子代理**：
   - 并行集中的所有组同时启动子代理
   - 串行链中的组在前置组完成并通过验收后启动
   - 每个子代理的 prompt 中传递其派发包的全部信息

6. **追踪完成状态**：
   - 每个组完成后立即验证其验收标准，不等待其他组
   - 记录已完成组的验收证据
   - 阻塞的组显式标记阻塞原因

### 子代理派发规范

**必须传入的信息类型**：

- **组范围**：该组负责的组件/模块集合及其边界定义
- **任务序列**：有序的实现任务列表，每个任务附带其设计追溯链
- **需求映射**：该组覆盖的 FR-ID 列表，每个 FR 附功能描述（从 g102 提取）
- **组件定义**：该组每个组件的细化定义（类名、成员变量、方法签名、目标文件路径）
- **接口定义**：该组提供和消费的接口定义（参数签名、调用约定、协议绑定）
- **数据对象定义**：该组涉及的数据对象定义（字段名、类型、约束）
- **契约绑定**：该组必须遵从的契约制品列表，区分提供方契约和消费方契约
- **设计参考**：架构设计文档路径及章节
- **验收标准**：量化的通过/失败条件
- **执行引擎**：推荐的引擎类型及其验证模式
- **前置产出**（条件性）：已验证的前置组产出位置

**必须产出的信息类型**：

- **实现证据**：产出的代码、配置、数据库迁移脚本等
- **验证证据**：执行的验证命令及其退出码/结果摘要
- **契约合规自检**：对每个契约绑定，声明该组的实现是否满足契约要求，并附验证方式
- **组件-定义一致性自检**：验证实现代码中的类名、方法签名、成员变量与组件定义文件一致
- **需求覆盖自检**：对需求映射中的每个 FR-ID，声明是否已实现并指明实现位置（文件路径+行号范围）
- **阻塞项记录**：任何阻止完成的问题，含阻塞条件和建议解决方案

### 执行引擎路由规则

根据子计划的类型标注，路由到对应的执行模式：

- 基础设施类（环境搭建、构建配置、数据库迁移）→ 命令执行验证模式
- 功能实现类（业务逻辑、算法、API 实现）→ TDD 验证模式
- 集成类（跨组联调、端到端流程）→ 条件扩展验证模式

### 出口条件

- 所有执行组已派发（并行或串行）
- 所有已启动组通过其验收标准（或有显式阻塞记录）
- 每个完成组的契约合规自检 + 组件-定义一致性自检 + 需求覆盖自检已提交
- 未完成的组有明确的阻塞原因和解除条件

## Phase 2: 契约合规验证

### 目标

集中验证三层一致性：
1. **接口层**：跨组接口的提供方和消费方是否均满足契约制品
2. **组件层**：实现代码中的类定义、方法签名、成员变量是否与组件定义文件一致
3. **数据层**：实现代码中的数据对象是否与数据对象定义文件一致

### 行为规则

1. **验证跨组契约双边合规**：对 `artifacts/implementation/contracts/` 下的每个契约制品（g402 产出）：
   - 检查提供方实现是否产出了契约约定的输出（格式、字段、错误语义）
   - 检查消费方实现是否按契约约定的方式消费（协议、序列化、错误处理）
   - 依据是 g402 产出的契约制品文件

2. **验证组件实现与定义一致性**：对 `artifacts/implementation/component-defs/` 下的每个组件定义（g402 产出）：
   - 检查实际代码中的类名是否与组件定义文件一致
   - 检查公共方法签名（方法名、参数名/类型、返回值类型）是否与定义一致
   - 检查成员变量（名称、类型、可见性）是否与定义一致
   - 依据是 g402 产出的组件定义文件

3. **执行跨组集成断言**：对每个跨组契约，至少有一条集成断言从消费方发起，验证提供方的实际响应与契约定义一致。

4. **产出契约合规矩阵**：

   | 契约制品/定义文件 | 提供方合规 | 消费方合规 | 实现一致性 | 证据 |
   |------------------|-----------|-----------|-----------|------|
   | （文件路径） | PASS/FAIL/UNTESTED | PASS/FAIL/UNTESTED | PASS/FAIL/UNTESTED | （验证方式或失败原因） |

5. **处理合规失败**：
   - FAIL → 标记对应的执行组，要求修复后重新验证
   - UNTESTED → 补充验证方式后重新评估
   - 所有条目必须收敛到 PASS 才能退出

### 出口条件

- 契约合规矩阵中每个条目的提供方和消费方均为 PASS
- 组件实现-定义一致性检查全部 PASS
- 无 FAIL 或 UNTESTED 条目
- 所有跨组集成断言通过

## Phase 3: 需求覆盖率验证

### 目标

验证所有需求条目均有对应的代码实现，无遗漏需求、无孤儿代码，全链可追溯。

### 行为规则

1. **建立 FR→代码全链追溯矩阵**：

   对 g100 需求基线中的每个 FR-ID，逐项验证：

   | FR-ID | 功能描述 | 设计依据 | g402 定义 | 实现文件 | 覆盖状态 | 证据 |
   |-------|---------|---------|-----------|---------|---------|------|
   | FR-001 | （从 g102 提取） | G301/G302/G303 章节 | data-defs/interface-defs 文件 | 代码文件路径+行号 | COVERED/MISSING | Phase 1 自检结果 |

   - 设计依据从实施计划的追溯链中提取
   - g402 定义从子代理的需求覆盖自检中提取
   - 实现文件通过代码审查或自动化 grep 验证

2. **MVP 完整性检查**：

   对 g103 MVP 定义中的每个 MVP-ID，检查其所有关联 FR 是否均已实现：

   | MVP-ID | 关联 FR | 已覆盖 FR | 未覆盖 FR | MVP 完成度 |
   |--------|---------|----------|----------|-----------|
   | MVP-001 | FR-001, FR-002, FR-003 | FR-001, FR-002 | FR-003 | 66% |

3. **孤儿代码检测**：

   - 扫描所有实现代码目录，提取所有公开类型/函数/接口
   - 与 g402 定义文件交叉对比
   - 标记不在任何组件定义或接口定义中的公开符号
   - 产出一份孤儿代码清单（如有），记录：文件路径、符号名、建议处理方式

4. **全链追溯完整性检查**：

   验证以下追溯链每段都不间断：

   ```
   FR-ID → 设计文档章节 → g402 定义文件 → 组件/接口定义 → 实现代码
   ```

   每段断裂都标记为 `MISSING_LINK`，记录断裂位置。

### 出口条件

- 追溯矩阵中所有 FR-ID 的覆盖状态均为 COVERED
- 所有 MVP 的完成度为 100%
- 无孤儿代码（或孤儿代码已显式标记为"有意保留"并附理由）
- 全链追溯无 MISSING_LINK

### 处理验证失败

| 失败类型 | 处理方式 |
|----------|----------|
| FR 未覆盖 | 标记对应的执行组，创建补齐任务 |
| MVP 未完成 | 列出未覆盖 FR，按优先级排序，建议补齐策略 |
| 孤儿代码 | 逐项判定：追溯缺失（补追溯链）/ 冗余代码（标记删除）/ 有意保留（记录理由） |
| 追溯链断裂 | 定位断裂点，补充缺失的中间环节 |

### 与上游技能的协作

**与 g402-implementation-refinement**：
- 消费其产出（`artifacts/implementation/` 下的全部细化定义）作为编码权威来源
- Phase 1 的派发包直接从 g402 产出中提取组件定义、接口定义、数据对象定义
- Phase 2 的验证依据为 g402 的产出文件

- 从子计划的追溯链中提取 FR→设计→任务的映射关系

## 关键原则

| 原则 | 说明 |
|------|------|
| **设计细化前置** | 编码级定义由 g402 在编码前完成。组件类名、方法签名、接口参数、数据字段全部冻结后再派发编码 |
| **需求驱动编码** | 每个执行组知道自己在为哪些 FR 编码，每个 FR 都能追溯到实现代码 |
| **分组隔离** | 每个执行组收到自包含的派发包（g402 定义 + 子计划 + 需求映射 + 契约），组间无共享可变状态 |
| **逐 Phase 验证** | 每个 Phase 有显式出口条件，前一 Phase 不通过，后一 Phase 不启动 |
| **依赖图驱动** | 并行/串行的判定来源于主计划的依赖图，不自行假设分层结构 |
| **消费而非产出定义** | 本技能消费 g402 的细化定义、plan-generation 的计划、g100 的需求，不创建新的定义文件或需求文件 |
| **三层验证闭环** | 契约合规（Phase 2）+ 需求覆盖（Phase 3）形成完整质量闭环，任一未通过不交付 |

## 常见问题

**Q: 如果 g402 细化定义尚未完成怎么办？**
A: 本技能的前置条件是 g402-implementation-refinement 已完成。如 `artifacts/implementation/` 为空或缺失关键子目录，先执行 g402。

**Q: 如果 g100 需求基线不存在怎么办？**
A: 本技能将需求基线列为必选输入。如 `artifacts/requirements/003-requirements-baseline.md` 不存在，中断并要求先执行 g100 需求阶段（g102）。

**Q: 如果实施计划中没有显式标注分组信息和依赖图怎么办？**
A: 此时应回到 `implementation-plan-generation`，要求其产出包含依赖图和分组信息的计划。本技能不补全计划生成阶段的缺失。

**Q: Phase 1 派发的子代理失败了怎么办？**
A: 记录失败原因和阻塞条件。如果是契约冲突，回推到 g402 重新协商契约。如果是组件定义与实现不一致，修复后重新派发该组。其他独立组不受影响继续执行。

**Q: Phase 3 发现 FR 未覆盖怎么办？**
A: 标记对应的执行组，创建补齐任务。如果 FR 确实不需要实现（如被 MVP 范围裁剪），在追溯矩阵中显式标注 `NOT_IN_SCOPE` 并附理由。所有未覆盖 FR 必须有明确的终止状态。

**Q: 细化定义的权威来源是什么？**
A: g402-implementation-refinement 产出的 `artifacts/implementation/` 下的文件是唯一权威来源。出现冲突时以 g402 定义为准；若 g402 定义与 G301/G302/G303 架构设计冲突，以架构设计文档为准。

**Q: layered-dev 和 layered-parallel-implementation 是什么关系？**
A: `layered-parallel-implementation` 是本技能的早期轻量版本（仅 2 Phase）。本技能（layered-dev）是完整版本，增加 g100 需求输入、需求追溯注入和 Phase 3 需求覆盖率验证。优先使用本技能。
