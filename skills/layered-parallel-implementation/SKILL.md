# 分层并行实施设计

## 目标

将已完成详细设计的复杂软件系统，转化为可并行执行的实施计划。核心思路：

> 按技术边界拆分 → 独立子代理分派 → 主代理整合 → 用户锁定方案

## 执行总流程

```
前提确认 ──→ 策略选择 ──→ 分层设计（子代理） ──→ 并行排期 ──→ 协调机制 ──→ 设计固化
```

## 阶段一：前提确认

### 目标
在提出任何实施策略之前，确认关键前提是否满足。

### 行为规则

1. **盘点上游设计资产**：确认设计文档完整性。如果发现核心文档缺失，先暂停并请求补充。

2. **确认团队配置**：用多选题询问用户团队技能栈。一次一个问题，从最重要的维度开始。

   问法示例：
   > 当前团队配置如何？
   > A. 全栈团队（有不同技术栈的工程师，可并行开发）
   > B. 后端为主（主要后端工程师，前端依赖外部）
   > C. 算法为主（有算法工程师，应用层开发资源有限）
   > D. 小团队/个人（需按关键路径逐步推进）

3. **验证基础设施**：确认开发环境、硬件、容器平台等就绪状态。

4. **标记已知约束**：将设计评审中遗留的阻塞问题显式列出。

### 出口条件
- 团队配置已通过用户选择锁定
- 关键约束和风险已显式记录
- 所有前提条件有明确的"就绪/等待"状态

---

## 阶段二：策略选择

### 目标
呈现 2-3 种互斥的实施策略，等待用户选择。不自动推进。

### 三种标准策略模板

**策略A：分层并行（推荐默认）**

| 维度 | 描述 |
|---|---|
| **拆分方式** | 按技术栈边界（算法层/业务层/表现层） |
| **并行线** | 每层独立团队并行推进 |
| **集成时机** | 后半段集中联调 |
| **适合场景** | 全栈团队，技术栈分层清晰 |
| **优势** | 资源利用率最高，进度可控 |
| **风险** | 接口变更需要跨层同步 |

**策略B：场景驱动**

| 维度 | 描述 |
|---|---|
| **拆分方式** | 按业务流程（建档→跟踪→展示→管理） |
| **并行线** | 每次交付一个完整场景 |
| **集成时机** | 每个场景内自包含集成 |
| **适合场景** | 需要快速出可演示成果 |
| **优势** | 每阶段有可见产出，风险早暴露 |
| **风险** | 存在重复集成开销 |

**策略C：基础设施优先**

| 维度 | 描述 |
|---|---|
| **拆分方式** | 先技术骨架再业务填充 |
| **并行线** | 前1-2周集中建基础设施 |
| **集成时机** | 基础设施就绪后业务并行 |
| **适合场景** | 长期演进项目，基础设施复杂 |
| **优势** | 技术债风险低，适合长期项目 |
| **风险** | 前期产出不可见 |

### 行为规则
- 同时呈现三个选项，给出简要对比
- 标注推荐选项（通常是策略A）
- 等待用户明确选择后才进入下一阶段

---

## 阶段三：分层设计（核心）

### 目标
将系统按技术边界拆分为独立层，每层有清晰的契约。

### 子代理执行模式

这是本技能的核心机制：

```
主代理                          子代理集群
  │                               │
  ├─ 盘点系统组件，判定层数 ────────┤
  │                               │
  ├─ 派发 Layer 1 ──────────────→ SubAgent A
  │    (传递：组件清单、层定义模板)   │  产出：layer-1-design.md
  │                               │
  ├─ 派发 Layer 2 ──────────────→ SubAgent B
  │    (传递：组件清单、层定义模板)   │  产出：layer-2-design.md
  │                               │
  ├─ 派发 Layer N ──────────────→ SubAgent N
  │    (传递：组件清单、层定义模板)   │  产出：layer-N-design.md
  │                               │
  ├─ 等待所有子代理完成 ───────────┤
  │                               │
  ├─ 验收各层产出 ─────────────────┤
  │   - 检查层定义模板填充完整       │
  │   - 检查接口契约五要素齐全       │
  │   - 检查无循环依赖              │
  │                               │
  ├─ 整理合并为统一设计文档 ────────┤
  │                               │
  └─ 输出待用户审查 ──────────────┘
```

#### 子代理文件存储规范

每层独立目录：
```
docs/implementation/
├── layer-1-<name>/
│   └── layer-1-design.md
├── layer-2-<name>/
│   └── layer-2-design.md
├── layer-N-<name>/
│   └── layer-N-design.md
└── consolidated-design.md    # 主代理合并产出
```

#### 子代理启动参数模板

```yaml
subagent_params:
  skill: layered-parallel-implementation
  description: "Layer N: <层名称> 分层设计"
  prompt: |
    你是 Layer N 的分层设计执行子代理。

    ## 任务
    为以下组件集合制定该层的详细设计。

    ## 层上下文
    - 层编号: Layer N
    - 层名称: <名称>
    - 技术栈: <语言/框架>
    - 所属边界: <边缘/中心/客户端>

    ## 负责组件
    - <组件ID>: <组件名称> — <职责>

    ## 上游依赖
    - <接口ID>: <上游层> → <本层> (<协议>)

    ## 下游承诺
    - <接口ID>: <本层> → <下游层> (<协议>)

    ## 设计要求
    1. 使用层定义模板填充所有字段
    2. 定义每个跨层接口的五要素（协议/格式/质量属性/生命周期/版本）
    3. 绘制本层内部模块依赖图
    4. 标注本层的关键风险和缓解方案
    5. 给出本层的时间线估算（骨架/核心/集成 三段）

    ## 输出
    - 文件: docs/implementation/layer-N-<name>/layer-N-design.md
```

### 分层原则

**规则1：按技术边界，不按业务功能**

```
正确: 数据层 → 算法层 → 业务层 → 表现层
错误: 用户模块 → 订单模块 → 报表模块（跨层混合）
```

判断标准：同一层内技术栈一致，层间通过标准协议通信。

**规则2：单层职责单一**

每层能用一句话描述：
> [层名称] 负责 [核心职责]，通过 [输入接口] 接收 [输入数据]，
> 经过 [处理逻辑]，通过 [输出接口] 输出 [输出承诺]。

如果描述中出现"和/同时/以及"，考虑进一步拆分。

**规则3：依赖单向无环**

```
允许:  Layer 1 → Layer 2 → Layer 3
禁止:  Layer 1 ↔ Layer 2（双向）
禁止:  Layer 3 → Layer 1（循环）
```

### 层定义模板

每层必须填充以下字段。子代理必须完整填写，主代理验收时逐项检查。

```markdown
## Layer N: [层名称]

### 技术定位
| 属性 | 值 |
|---|---|
| 所属边界 | [边缘/中心/客户端/混合] |
| 技术栈 | [语言/框架/运行时] |
| 部署形态 | [容器/进程/服务/静态文件] |
| 部署节点 | [具体部署位置] |

### 核心职责
[一句话描述，参考规则2]

### 输入契约
| 接口ID | 来源（层/系统） | 协议 | 数据格式 | 频率/量级 | 质量要求 |
|--------|-----------------|------|----------|-----------|----------|
| IN-XXX | ... | ... | ... | ... | ... |

### 输出契约
| 接口ID | 目标（层/系统） | 协议 | 数据格式 | 保证级别 | 质量承诺 |
|--------|-----------------|------|----------|----------|----------|
| OUT-XXX | ... | ... | ... | ... | ... |

### 内部组件清单
| 组件ID | 组件名称 | 职责摘要 | 关键算法/模式 | 优先级 |
|--------|----------|----------|---------------|--------|
| CMP-XXX | ... | ... | ... | ... |

### 内部依赖图
[使用 Mermaid 或文本 ASCII 图描绘组件间依赖]

### 时间线
| 阶段 | 时间段 | 里程碑 | 交付物 |
|------|--------|--------|--------|
| 骨架搭建 | Week X-Y | ... | ... |
| 核心实现 | Week Y-Z | ... | ... |
| 优化集成 | Week Z-N | ... | ... |

### Owner 与协作
| 角色 | 人数 | 职责 |
|------|------|------|
| ... | ... | ... |

### 关键风险
| 风险ID | 描述 | 概率 | 影响 | 缓解措施 |
|--------|------|------|------|----------|
| ... | ... | ... | ... | ... |
```

### 接口五要素定义

每个跨层接口必须明确定义以下五项：

```yaml
interface_contract:
  interface_id: "IFC-XXX"
  provider: "Layer N"
  consumer: "Layer N+1 / 外部系统"
  
  protocol:           # 要素1：协议
    type: [gRPC/MQTT/HTTP/共享内存/SQL]
    mode: [同步/异步/流式/批量]
    
  data:               # 要素2：数据格式
    request: "主要字段及类型（伪代码）"
    response: "主要字段及类型（伪代码）"
    error: "错误码和错误体格式"
    
  quality:            # 要素3：质量属性
    latency_p95_ms: 100
    throughput_rps: 1000
    reliability: "at-least-once / exactly-once"
    availability: "99.9%"
    
  lifecycle:          # 要素4：生命周期
    version: "v1.0"
    freeze_point: "Week N 结束"
    change_policy: "冻结后变更需跨层审批"
    
  error_semantics:    # 要素5：错误语义
    timeout_ms: 5000
    retry_policy: "指数退避，最大3次"
    fallback: "降级策略描述"
```

### 验证检查点

主代理验收各层产出时，逐层检查以下清单：

```
边界清晰性：
□ 层的输入是否全部来自明确的接口？
□ 层的输出是否全部通过明确的接口？
□ 层职责描述是否不包含"和/或"等模糊词汇？

接口完整性：
□ 每个跨层接口是否有唯一ID？
□ 每个接口是否定义了五要素？
□ 每个接口是否有明确的冻结日期？

依赖单向性：
□ 依赖图无循环？
□ 无跨层跳跃依赖（Layer 1 直接调 Layer 3）？

可实现性：
□ 单层核心开发周期 <= 4周？
□ 每层有明确 Owner？
□ 技术栈与团队技能匹配？

可测试性：
□ 输入可模拟？
□ 输出可验证？
□ 有契约测试计划？
```

### 常见错误模式

| 模式 | 症状 | 纠正 |
|------|------|------|
| **层职责过大** | 描述超过 50 字，包含多个动词 | 按技术栈边界再拆分 |
| **接口模糊** | 只写"JSON通信" | 补充 JSON Schema、协议、错误格式 |
| **隐性依赖** | Layer 2 直接读数据库 | 所有数据流必须过层间接口 |
| **过早优化** | Week 1 就定义缓存策略 | 骨架阶段先跑通，优化放 Week 5+ |

详细的分层设计指南见 `references/layer-design-guide.md`。

## 阶段四：并行排期

### 目标
将各层时间线组织为整体甘特图。

### 行为规则

1. **三阶段节奏**：每层按"骨架→核心→集成"三段式推进
2. **接口冻结点**：架构核心接口在早期冻结（Week 1-2）
3. **缓冲预留**：最后 1-2 周专门用于集成测试

### 时间线模板

```
            Week 1    Week 2    Week 3    Week 4    Week 5    Week 6
Layer 1:    [骨架]    [核心────] [核心──] [集成]    [优化]
Layer 2:    [骨架]    [核心────] [核心──]          [集成]
Layer 3:    [骨架]    [核心────────────] [集成]
QA:                           [契约测试] [集成测试] [性能测试] [验收]
                             ↑ 冻结点
```

## 阶段五：协调机制

### 目标
定义跨层协作规则，确保层间同步有序。

### 必须定义的机制

**周期同步**：
```
层级       参与者           频率    内容
层内      各层Owner+成员     每日   本日进度/明日计划/阻塞项
跨层接口  相邻层Owner        按需   接口变更协商
全团队    所有Owner          每周   进度汇报/风险识别/决策升级
```

**接口变更控制**：

冻结前（Draft 阶段）：自由修改，知会下游层即可
冻结后（Stable 阶段）：变更需提交申请，包含：
- 变更内容与原因
- 影响范围评估（哪些层受影响）
- 回滚方案
- 下游层 Owner 审批

**技术决策升级**：
```
层内技术选型 → 层 Owner 决策（24h 内）
跨层接口变更 → 架构师审批（48h 内）
阻塞性决策   → 项目经理升级（立即）
```

## 阶段六：设计固化

### 目标
将各层设计合并为统一实施文档，通过用户审批。

### 行为规则

1. **合并各层产出**：主代理将各子代理产出的 `layer-N-design.md` 整合为 `consolidated-design.md`
2. **自检清单验证**：检查所有验证检查点通过
3. **用户审批门**：逐段呈现设计，等待用户确认
4. **生成实施计划**：用户批准后，调用 `writing-plans` 技能创建详细实施计划

### 设计文档结构

```
docs/
├── superpowers/
│   └── specs/
│       └── YYYY-MM-DD-implementation-phase-design.md   # 整体设计
├── implementation/
│   ├── layer-1-<name>/
│   │   └── layer-1-design.md    # Layer 1 详细设计
│   ├── layer-2-<name>/
│   │   └── layer-2-design.md    # Layer 2 详细设计
│   ├── layer-N-<name>/
│   │   └── layer-N-design.md    # Layer N 详细设计
│   └── consolidated-design.md   # 主代理合并产出
└── superpowers/
    └── plans/
        └── YYYY-MM-DD-implementation-plan.md           # 实施计划
```

### 出口条件
- 所有层的 `layer-N-design.md` 已产出
- 自检清单通过
- 用户批准设计
- 实施计划已生成（`writing-plans` 产出）

---

## 关键原则

1. **逐问题推进**：一次只问一个问题，避免信息过载
2. **多选题优先**：提供 A/B/C/D 选择，降低用户沟通成本
3. **增量验证**：每个阶段结束时等待用户确认
4. **子代理独立**：每层完全独立，子代理之间不共享状态
5. **主代理把关**：主代理负责验收、整合、质量检查
