## 核心原则

`plan_tracker.md` 中手写标记的"完成"不可轻信。必须通过独立的验收报告进行交叉验证。

本skill的目标：**在状态被确认前，用客观报告拦截任何未达标的"完成"声明。**

## 评审触发时机

以下情况必须激活本skill：

1. 用户声明"Plan X 已完成"或"任务做完了"
2. 用户要求"检查当前计划能否标记完成"
3. 用户要求"评审执行结果"或"验收一下"
4. 任何 Task 状态拟从"执行中"改为"完成"或"部分完成"时
5. 用户询问"这个Task能算完成了吗"

## 与计划就绪评审的分工

| 阶段 | 职责 | Skill |
|------|------|-------|
| 计划编写后 | 检查计划本身的验收标准是否清晰 | plan-readiness-review |
| 计划执行后 | 检查实际产出是否达到计划中的标准 | plan-completion-gate-review（本skill） |

**plan-readiness-review 不能替代本skill。** 即使计划写得再好，执行后也必须经过完成门禁。

## 执行工作流程

### Step 1: 读取追踪文件

读取 `docs/exec-plans/plan_tracker.md`，提取：
- 当前活跃 Plan 及其所有 Task
- 每个 Task 的声明状态（待执行 / 执行中 / 部分完成 / 完成 / 阻塞中）
- 每个 Task 附带的验收报告路径（如有）

**如果用户未指定Plan**：从 plan_tracker.md 中找出最近被更新的 Plan 作为默认目标，并向用户确认。

### Step 2: 收集验收报告

对每个已标记"完成"或"部分完成"的 Task：
- 检查报告中引用的验收报告文件是否存在（路径相对于 `docs/exec-plans/`）
- 如报告路径为空或文件不存在，记录为"报告缺失"

**报告缺失的处理**：该 Task 自动视为"无法验证"，不得标记为"完成"或"部分完成"。

### Step 3: 解析验收结果

读取每个存在的验收报告（JSON格式），提取：
- `summary.p0_total` / `summary.p0_pass`
- `summary.p1_total` / `summary.p1_pass`
- `summary.p2_total` / `summary.p2_pass`
- `details` 中各条目的 `priority`、`status`、`dimension`
- `details` 中 `status` 为 `ENV_GAP` 的条目

### Step 4: 状态一致性校验

将实际验收结果与状态机规则对比，检查以下违规情况：

| 违规类型 | 判定条件 | 处理 |
|----------|----------|------|
| 假完成 | 声明"完成"但 P0 未全部通过 | 强制降级为"阻塞中" |
| 跳级 | 从"执行中"直接标记"完成"（跳过了部分完成） | 要求回退到正确状态 |
| P1 隐瞒 | 声明"完成"但 P1 有未清零缺口 | 降级为"部分完成" |
| ENV_GAP 误报 | P0 维度存在 ENV_GAP 但状态不是"阻塞中" | 强制降级 |
| 应降未降 | 已标记完成但后续报告显示 FAIL | 触发降级 |

### Step 5: 环境缺口（ENV_GAP）评估

对报告中 `status` 为 `ENV_GAP` 的条目：

| 优先级 | 规则 |
|--------|------|
| **P0** | 不可豁免。若 P0 存在 ENV_GAP，Task 不可标记完成或部分完成，必须先解决环境或找到替代验证方式。用户明确书面批准的例外除外。 |
| **P1** | 可记录为 ENV_GAP，但必须在**下一 Plan 启动前**补验，否则升级为 P0 阻塞。 |
| **P2** | 可记录为 ENV_GAP，挂技术债追踪，不强制补验时间。 |

### Step 6: 输出门禁评审报告

## 状态判定规则（铁律）

以下规则不可违反：

```
IF 报告缺失 → "无法验证"（不可标记完成/部分完成）
IF P0_pass < P0_total → "阻塞中"
IF P0_pass == P0_total AND P1_pass == P1_total AND 无P0 ENV_GAP → "完成"
IF P0_pass == P0_total AND (P1_pass < P1_total OR 存在P1 ENV_GAP) AND 缺口已登记 → "部分完成"
IF 状态从"执行中"直接跳到"完成"（且P1未全过）→ 违规，回退到"部分完成"或"阻塞中"
```

**门禁铁律：**
1. **先验证后声称完成**：任何 Task 在标记完成前，必须先有验收报告。
2. **P0 失败 = 阻塞**：任何 P0 维度未通过，Task 不可标记完成或部分完成。
3. **P0 环境缺口不可豁免**：P0 的 ENV_GAP 与 FAIL 同等处理，除非用户明确书面批准。
4. **状态不可跳级**：不允许从"执行中"直接跳到"完成"。
5. **降级触发**：后续验收发现之前通过的项现在 FAIL，Task 必须降级。

## 禁止行为

1. **不可直接采信 plan_tracker.md 的手写状态**——必须以验收报告为唯一判定依据。
2. **不可将 ENV_GAP 视为 PASS**——ENV_GAP 是独立状态，P0 的 ENV_GAP 按 FAIL 处理。
3. **不可将 P0 FAIL 视为"部分完成"**——P0 有任何失败即为"阻塞中"。
4. **不可跳过报告缺失的 Task**——报告缺失等同于无法验证，必须指出。
5. **不可在报告缺失时推测结果**——没有报告就不能猜测"应该没问题"。

## 示例

### 输入场景

用户说："Plan B 完成了，帮我看一下。"

### 执行过程

1. 读取 `docs/exec-plans/plan_tracker.md`，找到 Plan B 的 Task 列表
2. Task 3 标记为"完成"，引用报告 `reports/B-Task3-20260604.json`
3. 读取报告：summary 显示 p0_pass=5, p0_total=5, p1_pass=2, p1_total=2
4. 状态判定：P0全绿 + P1全绿 → 符合"完成"条件
5. Task 4 标记为"完成"，引用报告 `reports/B-Task4-20260604.json`
6. 读取报告：summary 显示 p0_pass=3, p0_total=4（D1覆盖率未达标）
7. 状态判定：P0未全绿 → 违规，应降级为"阻塞中"

### 输出片段

```markdown
## 关键阻塞项

| Task | 维度 | 问题 | 建议 |
|------|------|------|------|
| Task 4 | D1 | P0覆盖率未达标（实际51.3%，门槛60%） | 补充测试覆盖至门槛以上 |

## 状态修正建议

| Task | 当前声明状态 | 建议状态 | 原因 |
|------|-------------|----------|------|
| Task 4 | 完成 | 阻塞中 | P0维度存在FAIL（D1覆盖率） |

**门禁结论：阻塞**——存在P0失败项，Plan B 不可标记为完成。
```
