  在执行任何多步骤计划（Plan）之前，必须先评审计划本身的就绪质量。
  使用此skill检查：Task验收标准是否可量化、关键质量维度是否覆盖、
  依赖关系是否合理、状态升级路径是否清晰、风险预案是否完备、每个Task是否可追溯到设计与MVP需求。

  "这个计划准备好了吗"、写完计划文件后要求评估、或计划中存在模糊的"编译通过即可"
  "无错误"等弱验收表述。即使计划看起来完整，也要主动检查是否有隐藏的遗漏维度。
  任何包含多个Task的实施计划，在执行前都应经过此评审。
---

## 核心原则

未经验评审的计划是技术债的源头。计划阶段的模糊验收标准（如"编译通过即可""无错误"）会在执行阶段被放大为真实的遗漏。

本skill的目标：**在编码开始前，消灭计划中的模糊性、遗漏项和不可验收的Step。**

## 评审触发时机

以下情况必须激活本skill：

1. 用户要求"执行下一阶段任务"或"按计划执行"
2. 用户要求"评审这个计划""检查计划完整性"
3. 用户写完新的计划文件（如 `docs/superpowers/plans/*.md`）
4. 用户询问"这个计划准备好了吗"
5. 计划中存在模糊的验收表述（见下方"模糊表达清单"）

## 评审维度

对每个Plan的每个Task进行以下8个维度的审查：

### D1 可验收性 — 每个Step是否有明确的验收标准

**合格标准：** 每个Task的每个Step都必须回答"怎样才算完成"。

**检查方法：**
- 扫描计划中每个Task下的Step
- 判断是否有明确的"验证"或"验收"段落
- Step中是否包含"Run:""Expected:""验证"等关键词

**常见问题：**
- Step只写"创建文件"但没有验证文件内容是否正确
- Step只写"编写测试"但没有验证测试是否通过
- Step写"配置环境"但没有验证环境可用

### D2 可量化性 — 验收标准是否包含具体数字门槛

**合格标准：** 关键验收标准必须包含可测量的数字。

**检查方法：**
- 扫描"Expected:"段落
- 判断是否包含具体数值：百分比（如 >=60%）、时间（如 <50ms）、数量（如 0 error）、频率（如 1Hz±50ms）

**常见问题：**
- "编译通过即可" → 缺少 warning 数量门槛
- "无错误" → 缺少具体的错误类型和数量
- "性能良好" → 缺少 P99/TPS 数值
- "测试通过" → 缺少覆盖率门槛

### D3 完整性 — 是否覆盖了关键质量维度

**合格标准：** 计划应覆盖以下至少4个关键维度（根据项目类型调整权重）：

| 维度 | 典型验收点 | 常见遗漏 |
|------|-----------|----------|
| 测试覆盖 | 单元测试、集成测试、覆盖率门槛 | 只写"编写测试"不写覆盖率 |
| 安全检查 | 密钥扫描、依赖漏洞、权限校验 | 骨架阶段完全忽略安全 |
| 编译/构建 | 零 warning、零 error、类型检查 | "编译通过即可" |
| 集成断言 | API响应验证、数据一致性、端到端 | 只验证单个组件 |
| 性能基准 | P99延迟、TPS、帧率稳定性 | 骨架阶段跳过性能 |
| 契约一致 | Proto/OpenAPI/Mock与实现一致性 | 前后端契约未对齐 |

**检查方法：**
- 为每个维度在计划中搜索对应关键词
- 统计覆盖的维度数量
- 标记完全缺失的维度

### D4 依赖合理性 — Task之间的依赖是否有遗漏

**合格标准：** 每个Task的前置依赖必须显式声明，且无循环依赖。

**检查方法：**
- 检查每个Task的"前置条件"或"依赖"段落
- 验证依赖的Task是否在计划中确实存在
- 检查是否存在"隐式依赖"（如Task 5依赖Task 3的中间产物但未声明）

**常见问题：**
- 并行Task之间共享资源但未声明竞态
- 后续Task依赖前面Task的特定输出格式但未声明
- 缺少"Schema Freeze""API Freeze"等关键检查点

### D5 状态清晰度 — 状态定义和升级条件是否明确

**合格标准：** 计划中必须定义Task/Plan的状态和升级规则。

**检查方法：**
- 检查是否定义了状态（如 待执行/执行中/完成/阻塞）
- 检查是否有明确的"完成"定义
- 检查降级触发条件（如发现新问题后如何处理）

**常见问题：**
- 只有"完成"和"未完成"两态，没有"部分完成"或"阻塞"
- "完成"定义模糊（如"代码写完即可"）
- 缺少降级机制（已完成Task因代码变更重新FAIL如何处理）

### D6 风险预案 — 是否有环境缺口和降级方案

**合格标准：** 计划必须预见到可能的执行阻碍并给出预案。

**检查方法：**
- 检查是否有"环境缺口""降级方案""回退"等段落
- 检查Windows/Linux/macOS差异是否被考虑
- 检查工具缺失时的替代方案

**常见问题：**
- 假设所有工具都已安装（如 actionlint、gosec、Playwright）
- 假设测试环境始终可用（如数据库、Docker）
- 没有回退方案（如 POC 失败后的 B 计划）

### D7 可追踪性 — 是否有明确的追踪格式和更新规则

**合格标准：** 计划的执行进度必须能被客观追踪，不依赖人的记忆。

**检查方法：**
- 检查是否指定了追踪文件（如 plan_tracker.md）
- 检查追踪格式是否包含：状态、证据链接、缺口记录
- 检查是否有禁止的行为（如手动改"完成"而不附证据）

**常见问题：**
- 计划文件是唯一的追踪手段，没有单独的进度文件
- 状态只有"完成"，没有"部分完成"
- 缺少完成声明模板（需要什么证据才能标记完成）

### D8 业务可追溯性 — 每个计划项是否可追溯到设计文档与 MVP 需求

**合格标准：** 任何实现项都是为业务服务。每个 Task 必须能回答"为哪个业务需求服务"，形成完整的追溯链：Task → 设计文档 → MVP 需求。

**为什么重要：** 三个月后有人问"这段代码为什么这么写"，能沿着计划→设计→需求一路追溯到原始业务决策。没有追溯链的任务，只是技术自嗨，不是合格的工程计划。

**检查方法：**
- 扫描每个 Task，检查是否包含设计追溯引用（设计文档路径 + 章节号）
- 扫描每个 Task，检查是否包含需求追溯引用（MVP ID 或需求 ID）
- 统计无任何追溯引用的 Task 数量（孤儿任务）
- 反向检查：需求文档/MVP 定义中的每项需求，是否在计划中至少有一个 Task 覆盖（孤儿需求）

**合格门槛：**
- 追溯链覆盖率 = 含追溯引用的 Task 数 / Task 总数，必须 = 100%
- 孤儿需求数 = MVP 需求项中无 Task 覆盖的数量，必须 = 0
- 每个追溯引用中的设计文档路径必须能定位到实际文件
- 每个追溯引用中的需求 ID 必须能在需求文档中 grep 到

**常见问题：**
- Task 只描述技术动作（"创建数据库表""编写 API"）但未说明服务于哪个业务需求
- 计划中只有技术任务列表，没有到需求/MVP 的反向链接
- 存在"技术自嗨"任务：纯粹的技术优化或重构，当前 MVP 阶段不需要
- 需求文档中有明确定义的 MVP 项，但计划中完全没有对应的实现 Task
- 追溯引用格式不规范（如只写"见需求文档"而不写具体 ID），导致无法机械验证

**追溯链格式要求**（与 `implementation-plan-generation` 的追溯链格式对齐）：

```
> **追溯链** | 设计: <设计文档路径> §<章节号> | 需求: <需求ID列表>
```

- `设计` 指向设计文档的具体章节（如 `G301 §3.2.1`、`003-interface-design.md §4.5`）
- `需求` 指向需求 ID（如 `FCT-003`、`MVP-004`），多个需求用逗号分隔
- 设计文档路径使用相对于仓库根目录的路径

## 评审工作流程

### Step 1: 读取计划文件

读取用户提供的计划文件（如 `docs/superpowers/plans/*.md`）和相关的追踪文件（如 `docs/exec-plans/plan_tracker.md`）。

如果用户没有指定计划文件，询问用户需要评审哪个计划。

### Step 2: 逐Task逐维度审查

对每个Task的每个Step，按D1~D8逐一审查。记录：
- **通过**：明确满足标准
- **警告**：存在风险但未严重到阻塞
- **失败**：不满足标准，可能导致执行遗漏
- **不适用**：该维度对当前Task无意义

### Step 3: 汇总评审报告

生成结构化评审报告，格式如下：

```markdown
# Plan 就绪评审报告 — <Plan名称>

## 评审概览

| 维度 | 通过 | 警告 | 失败 | 不适用 |
|------|------|------|------|--------|
| D1 可验收性 | X | Y | Z | W |
| D2 可量化性 | ... | ... | ... | ... |
| D3 完整性 | ... | ... | ... | ... |
| D4 依赖合理性 | ... | ... | ... | ... |
| D5 状态清晰度 | ... | ... | ... | ... |
| D6 风险预案 | ... | ... | ... | ... |
| D7 可追踪性 | ... | ... | ... | ... |
| D8 业务可追溯性 | ... | ... | ... | ... |

**总体评级**：READY / NEEDS_WORK / BLOCKED

## 关键阻塞项（必须修复才能执行）

| Task | Step | 维度 | 问题描述 | 建议修复 |
|------|------|------|----------|----------|
| Task 3 | Step 5 | D2 | "覆盖率 >= 60%" 缺少具体命令和输出格式 | 补充 `go test -coverprofile=...` 和 Expected 精确输出 |

## 警告项（建议修复但不阻塞）

| Task | Step | 维度 | 问题描述 | 建议修复 |
|------|------|------|----------|----------|
| Task 7 | Step 3 | D3 | 缺少性能基准验收 | 添加骨架阶段最低性能门槛或标注为P2 |

## 环境缺口预判

| 检查项 | 可能缺失的环境 | 建议预案 |
|--------|---------------|----------|
| `go test -race` | Windows 无 cgo | 在 CI/WSL 中补验，Plan 中标注 ENV_GAP |

## 执行建议

1. 先修复所有阻塞项
2. 再处理警告项
3. 确认环境缺口预案后，方可开始执行
```

### Step 4: 给出就绪评级

根据评审结果给出总体评级：

- **READY**：所有 D1~D8 维度无阻塞项，可以直接执行
- **NEEDS_WORK**：有警告项但无阻塞项，可以执行但建议先修复警告
- **BLOCKED**：有阻塞项，必须先修复才能执行

**禁止行为：** 即使只有1个阻塞项，也绝不允许评级为 READY 或 NEEDS_WORK。

## 模糊表达清单（自动检测关键词）

以下表达在计划中出现时，应自动标记为 D2 失败：

| 模糊表达 | 问题 | 建议替换为 |
|----------|------|-----------|
| "编译通过即可" | 忽略 warning | "编译零 warning：`go vet` 无输出" |
| "无错误" | 不具体 | "0 error，0 warning" |
| "测试通过" | 缺少覆盖率 | "测试 PASS 且覆盖率 >= X%" |
| "性能良好" | 无量化 | "P99 < Xms / TPS > Y" |
| "无异常" | 不具体 | "无 panic、无 DATA RACE" |
| "一致" | 不量化 | "字段名 100% 匹配 OpenAPI schema" |
| "稍后处理" | 无限期拖延 | "在 Task X 前必须完成" |
| "暂定" | 未确定 | 明确数值或标注为 ENV_GAP |
| "见需求文档" / "按需求实现" | 无具体需求 ID，无法追溯 | 标注具体需求 ID（如 FCT-003、MVP-004） |
| "优化性能" / "重构代码" | 无业务需求支撑，可能是技术自嗨 | 关联到具体 MVP 项或设计章节，或标注为非 MVP 范围 |

## 与执行门禁的关系

本skill（计划评审）与执行门禁skill是**互补关系**：

| 阶段 | 职责 | Skill |
|------|------|-------|
| 计划编写后 | 检查计划本身的质量：验收标准是否清晰、是否遗漏维度 | plan-readiness-review（本skill） |
| 计划执行中 | 检查实际产出是否达到计划中的标准 | plan-completion-gate（执行门禁） |
| 计划执行后 | 复盘执行与计划的偏差 | 复盘流程（项目级） |

**重要：** 即使计划评审通过（READY），执行时仍需逐Task运行执行门禁。计划评审不能替代执行验收。

## 示例

### 输入：一个存在问题的Plan片段

```markdown
### Task 3: 登录/登出 API + RBAC 中间件

- [ ] **Step 1: 编写 RBAC 中间件**
- [ ] **Step 2: 编写登录/登出 Handler**
- [ ] **Step 3: 编写登录测试**
- [ ] **Step 4: 注册路由并运行测试**
  Run: `go test ./internal/auth/... -v`
  Expected: `TestLoginHandler` PASS
- [ ] **Step 5: 登录接口响应时间**
  Run: Benchmark
  Expected: 性能良好
- [ ] **Step 6: 覆盖率**
  Run: `go test -coverprofile=...`
  Expected: 总覆盖率 >= 60%
- [ ] **Step 7: 提交**
```

### 输出：评审报告片段

```markdown
## 关键阻塞项

| Task | Step | 维度 | 问题描述 | 建议修复 |
|------|------|------|----------|----------|
| Task 3 | Step 3 | D1 | "编写登录测试" 没有验证测试是否通过 | 补充验证命令和Expected输出 |
| Task 3 | Step 5 | D2 | "性能良好" 无量化标准 | 替换为 "P99 < 50ms" 或 "Benchmark ns/op < X" |
| Task 3 | Step 6 | D2 | 覆盖率门槛合理，但缺少具体包级门槛 | 补充各关键包的最低覆盖率 |
| Task 1 | Step 1 | D8 | "创建数据库迁移脚本" 未说明服务于哪个业务需求 | 补充追溯链：`设计: G303 §2.1 \| 需求: MVP-001` |
| Task 7 | Step 2 | D8 | "优化查询性能" 无需求支撑，当前 MVP 阶段无对应业务项 | 关联到具体 MVP 项或标注为非 MVP 范围延后 |

## 警告项

| Task | Step | 维度 | 问题描述 | 建议修复 |
|------|------|------|----------|----------|
| Task 3 | Step 4 | D3 | 只验证了登录成功，缺少 RBAC 拒绝未授权请求的验证 | 补充 401/403 测试用例 |
| Task 5 | Step 2 | D8 | 需求 `MVP-007`（设备告警通知）在计划中无任何 Task 覆盖 | 补充覆盖该需求的 Task，或确认该需求已延后并登记 |
```

## 输出要求

1. **使用中文**：评审报告默认使用中文
2. **引用原文**：指出问题时必须引用计划中的原文片段
3. **给出建议**：每个问题必须附带具体的修改建议，不只是指出问题
4. **区分阻塞/警告**：明确告知用户哪些是必须先修复的，哪些可以后续优化
5. **禁止猜测**：如果不确定某个Step的意图，应标注"需要澄清"而不是假设
