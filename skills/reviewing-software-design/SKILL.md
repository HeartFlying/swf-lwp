# 软件设计评审

## 目标

使用本 Skill 对软件产品设计文档进行结构化、量化、可落地的开工准备度评审。评审必须回答一个问题：

> 研发工程师能否基于该设计文档开始产品代码开发，并且不需要反复澄清核心问题、不产生可预见的大规模返工？

评审前必须读取 `references/review-rubric.md`。如果被评审系统包含计算机视觉、YOLO、模型训练、微调、推理服务、模型部署、GPU/NPU/加速卡选型或算法验收指标，必须同时应用该参考文件中的 CV/算法评审部分。

## 评审角色

从以下角色视角评估文档。不要替缺席的干系人假设已确认；缺少证据时必须记录为问题。

| 角色 | 评审重点 |
|---|---|
| 产品负责人 | 业务目标、范围、用户流程、验收标准 |
| 领域专家 | 业务规则、术语、边界场景、监管或合规约束 |
| 系统架构师 | 系统边界、模块职责、集成方式、扩展性、兼容性 |
| 后端负责人 | API、事务、幂等、一致性、数据模型、迁移 |
| 前端/客户端负责人 | 页面流程、状态、权限、表单校验、错误态、API 易用性 |
| QA 负责人 | 可测试性、可量化验收、负向用例、回归范围 |
| SRE/运维 | 部署、容量、配置、观测、告警、回滚、灾难处理 |
| 安全/合规 | 认证、授权、数据隔离、隐私、审计、敏感数据保护 |
| 算法负责人 | 数据、标签、模型选择、训练、指标、推理、漂移 |
| 硬件/平台负责人 | GPU/NPU/加速卡选择、吞吐、延迟、成本、扩容 |

## 工作流程

1. **盘点输入**
   - 列出被评审文档、版本、日期、负责人和缺失文档。
   - 识别文档类型：产品方案、架构设计、概要设计、详细设计、API 设计、数据库设计、前端设计、部署设计、算法设计、模型训练方案或混合文档。
   - 判断是否适用 CV/算法评审。

2. **提取可实现主张**
   - 提取业务目标、范围内/范围外事项、核心流程、用户角色、业务规则、模块、集成点、数据实体、接口、页面/客户端状态、非功能目标、部署假设、验收标准。
   - 提取代码级可实现主张：代码分层与包结构、领域模型（实体/值对象/聚合根）、设计模式选择、模块交互时序与异步边界。
   - 提取关键逻辑的实现模板：核心流程伪代码、幂等/事务/补偿的代码片段、统一错误处理与日志模式、Trace/Context 传递机制。
   - 提取测试与可测试性设计：单元测试策略、核心用例（含边界值与异常输入）、Mock/依赖注入方案、测试数据构造策略。
   - 对 CV/算法系统，额外提取检测类别、业务场景、数据集、标注规则、模型版本、训练计划、推理引擎、硬件假设、阈值策略和指标。

3. **构建评审范围**
   - 所有系统必须使用 `references/review-rubric.md` 中的通用分类。
   - 有前端/客户端交互时，必须检查前端/客户端可实现性条目。
   - 有 CV/算法内容时，必须增加 CV/算法分类。
   - 每个分类标记为 `适用`、`不适用` 或 `证据不足`。不能因为文档未提及某项就给高分；缺失项必须有“不适用”的理由，否则记录为问题。

4. **优先检查一票否决项**
   - 先检查通用否决项，再检查前端/客户端否决项和 CV/算法否决项。
   - 任一否决项触发时，结论必须是 `不通过`，除非用户明确要求仅做 advisory scoring。

5. **逐项评分**
   - 先对每个分类下的 review item 逐项判断，再给出分类总分。
   - 每个适用分类按 0-3 分评分：
     - `0`：缺失、空泛或仅有标题。
     - `1`：有提及但不可直接实现，研发需要大量澄清。
     - `2`：基本可实现，但仍有少量明确缺口。
     - `3`：清晰、完整、可量化，可直接转化为实现和测试。
   - 如果分类内任一核心 item 为 0，该分类最高只能得 1 分；如果该 item 属于否决项，必须触发否决。
   - 每个问题必须标记为 `阻塞` 或 `非阻塞`，并尽量引用章节、页码、标题或原文证据。

6. **计算量化结果**
   - 将分类评分按权重换算为 100 分。
   - 先应用否决项和阻塞项，再应用分数阈值。
   - 分数和问题严重度冲突时，采用更严格结论。

7. **输出评审报告**
   - 给出执行结论、分数、否决状态、阻塞问题数和最终开工准备度。
   - 先列阻塞问题，再列非阻塞问题。
   - 给出角色视角评审意见。
   - 给出整改计划，必须包含 owner role、完成条件和验收证据。

## 评分权重

默认权重如下。用户提供项目权重时，以用户权重为准，但不得把“测试与验收”降到 10% 以下，除非用户明确说明该评审只关注架构建议而非开工准入。

| 分类 | 权重 |
|---|---:|
| 业务与范围 | 10% |
| 架构与模块 | 15% |
| 接口与数据 | 15% |
| 异常、可靠性与一致性 | 15% |
| 安全与合规 | 10% |
| 非功能与运维 | 5% |
| 测试与验收 | 15% |
| 工程落地与任务可拆解性 | 15% |

如果系统包含明显前端/客户端交互，将相关检查纳入“业务与范围”“接口与数据”“测试与验收”“工程落地与任务可拆解性”评分，不单独加权。

对于 CV/算法核心产品，必须为算法评审预留总分的 20%-30%。当算法性能是核心交付风险时使用 30%，并按比例压缩通用分类权重。

CV/算法评分拆分：

| 算法分类 | 算法分内部权重 |
|---|---:|
| 业务目标与类别定义 | 15% |
| 数据与标注 | 20% |
| 模型选择与训练计划 | 20% |
| 指标与验收 | 15% |
| 硬件与部署 | 15% |
| 监控、迭代与风险控制 | 15% |

## 结论规则

| 条件 | 必须结论 |
|---|---|
| 任一否决项触发 | 不通过 |
| 存在阻塞问题 | 暂缓并重新评审 |
| 分数 >= 85，且无否决项、无阻塞问题 | 通过 |
| 分数 70-84，且无否决项、无阻塞问题 | 有条件通过 |
| 分数 60-69，或存在重大未决歧义 | 暂缓并重新评审 |
| 分数 < 60 | 不通过 |

不要在存在阻塞问题时使用 `通过` 或 `有条件通过`。`有条件通过` 只表示存在可并行关闭的非阻塞问题；它不允许核心流程、核心 API、核心数据、权限边界、验收标准或关键异常处理仍不明确。

## 工程开工门槛

最终开工准备度必须按以下规则给出：

| Engineering readiness | 可执行含义 |
|---|---|
| Ready | 可正式开始产品代码开发。文档已覆盖核心流程、业务规则、API/数据、权限、异常、验收和落地任务，且提供了代码分层、领域模型、关键逻辑伪代码及测试策略，初级工程师可直接据此编码。 |
| Ready with conditions | 只允许启动不受缺口影响的低风险任务，例如脚手架、技术验证、非核心页面、基础配置；核心产品代码需等条件关闭。 |
| Not ready | 不应开始产品代码开发。最多进行 PoC、技术 spike、补充设计或评审整改。 |

只有在无否决项、无阻塞问题、验收标准可量化、API/数据/流程决策可追踪、代码分层与领域模型明确、关键逻辑附有实现模板时，设计文档才可以作为正式产品代码开发输入。

## 常见错误

| 错误 | 必须修正 |
|---|---|
| 把“提到了”当作“可实现” | 除非能直接转化为代码、配置、测试或验收，否则只能给 0 或 1 |
| 用平均分掩盖关键缺口 | 先应用否决项和阻塞项，再计算最终结论 |
| 只看架构图 | 必须同时检查数据、API、错误、权限、运维、测试和验收 |
| 忽略前端状态 | 有交互界面时必须检查加载态、空态、错误态、权限态、表单校验和 API 易用性 |
| 把 mAP 当作唯一算法验收 | 必须要求漏检率、误报率、延迟、FPS、分场景召回等业务指标 |
| 忽略文档缺失 | 缺少文档必须记录为证据缺口，不得默认已满足 |
| 只给建议不给决策 | 必须给出结论、分数、阻塞数量、整改计划和开工准备度 |
| 忽略代码结构 | 必须检查分层、包结构、领域模型和设计模式，确保架构意图能无损转化为代码 |
| 把"有接口定义"当作"能写代码" | 除非提供关键逻辑的伪代码或实现模板，否则研发对实现方式仍有分歧，初级工程师无法写出统一高质量代码 |
| 忽略可测试性设计 | 必须检查依赖注入、接口抽象和测试数据构造，确保单元测试可与业务代码并行开发 |

## 最终门禁

每次评审末尾必须输出：

```text
Engineering readiness: Ready | Ready with conditions | Not ready
Reason: [基于证据的简短理由]
```

## 在 G300 工作流中的调用契约

### 调用位置

`g300-detailed-design-entry` 的 `DD-08`

### 输入约定

必须消费以下产物：

- `artifacts/detailed-design/002-component-design.md`
- `artifacts/detailed-design/003-interface-design.md`（complete 模式）
- `artifacts/detailed-design/004-data-design.md`（complete 模式）
- `artifacts/detailed-design/005-design-review-validation.md`
- `artifacts/detailed-design/006-architecture-design-traceability.md`

### 模式差异

- `standard` 模式下，接口与数据分类下的详细设计评审项（如 API 契约完整性、数据模型状态机）标记为"不适用"并提供理由。
- `complete` 模式下全量评审。

### 输出约定

产出固定路径：

```
artifacts/detailed-design/007-engineering-readiness-review.md
```

### 下游消费

- `GS-Quality-Check`（DD-09）将其阻塞问题清单纳入质量检查范围。
- `GS-Review`（DD-10）将其作为必消费输入；评审通过条件要求 Engineering readiness ≠ `Not ready`。
- 下游 `implementation` 阶段通过 `handoff_summary` 获取开工准备度结论与整改计划。
