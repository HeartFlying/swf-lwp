# 设计评审细则

用于执行 `reviewing-software-design` 时的评分、否决和证据标准。

## 通用评审分类

| 分类 | Review items |
|---|---|
| 业务与范围 | 业务目标、明确范围、范围外事项、用户角色、用户流程、领域术语、业务规则、边界场景 |
| 架构与模块 | 系统边界、模块职责、依赖关系、集成方式、扩展性、迁移、兼容性、降级边界、代码分层与包结构、领域模型定义、设计模式选择、模块交互时序与异步边界 |
| 接口与数据 | API 契约、请求/响应示例、错误码、数据模型、状态机、一致性策略、数据迁移、幂等键、关键逻辑伪代码或实现模板、统一错误处理与日志模式 |
| 异常、可靠性与一致性 | 失败路径、重试、幂等、事务边界、补偿、降级、灾难处理、对账、人工介入 |
| 安全与合规 | 认证、授权、租户/数据隔离、敏感数据保护、审计、隐私、合规留痕 |
| 非功能与运维 | 性能目标、容量假设、部署拓扑、配置项、日志、指标、链路追踪、告警、回滚 |
| 测试与验收 | 功能验收、负向用例、性能验收、安全验收、数据校验、回归范围、上线验证、单元测试策略与核心用例、可测试性设计（依赖注入、Mock 策略、测试数据构造） |
| 工程落地与任务可拆解性 | 任务拆分、依赖顺序、模块到代码边界映射、配置开关、灰度路径、兼容策略、验收样例、代码入口与出口点、数据库迁移与回滚顺序、功能开关代码级定义 |

## 前端/客户端评审条目

有 Web、App、小程序、桌面端、CLI 或设备端交互时应用。

| 分类 | Review items |
|---|---|
| 用户流程 | 主流程、分支流程、返回/取消/重试、跨页面状态、权限差异 |
| 页面状态 | 加载态、空态、错误态、禁用态、成功态、部分成功态、长任务状态 |
| 输入与校验 | 表单字段、格式校验、服务端错误映射、重复提交、防抖、草稿或恢复策略 |
| API 易用性 | 前端所需字段、分页/排序/筛选、错误码可展示性、乐观更新或刷新策略 |
| 可测试性 | 页面验收标准、关键交互用例、可观测埋点、端到端测试边界 |

## 通用一票否决项

以下任一情况触发否决：

- 核心业务流程缺失。
- 关键业务规则为 TBD、相互矛盾或无法判定。
- 核心 API 契约缺失。
- 核心数据模型缺失。
- 权限边界不清楚。
- 关键路径缺少异常处理、事务边界或补偿策略。
- 验收标准缺失或不可量化。
- 无法判断设计如何拆解为工程任务，且该设计被要求作为正式开工输入。
- 代码分层结构、领域模型或核心模块到代码的映射缺失，导致研发无法确定类/包/接口的创建位置。
- 核心业务流程缺少伪代码或实现模板，存在两种及以上合理实现方式，初级工程师无法据此写出统一的高质量代码。

## 前端/客户端一票否决项

有交互界面且以下任一情况成立时触发否决：

- 核心用户流程或关键页面缺失。
- 权限差异会影响页面行为，但文档未定义。
- 核心表单或关键操作缺少校验、错误处理或重复提交策略。
- 前端依赖的核心字段、状态或错误码在 API/数据设计中缺失。
- 页面验收标准不可测试。

## 通用坏味道/好味道示例

| 领域 | 坏味道 | 好味道 |
|---|---|---|
| 业务目标 | 提升效率 | 异常订单人工处理量降低 40%；运营人员 3 分钟内完成判断 |
| 范围 | 支持用户管理及相关功能 | 本期支持创建、禁用、角色绑定；组织同步排除在本期外 |
| 业务规则 | 大额需要审批 | 金额 >= 100000 进入二级审批；VIP 客户跳过一级审批 |
| 接口 | 提供订单查询 API | 定义 URL、方法、请求、响应、错误码、分页、幂等、示例 |
| 状态 | 订单有待处理和完成状态 | 定义全部状态、允许转换、触发条件、禁止转换和补偿 |
| 可靠性 | 第三方失败时重试 | 超时 2s；重试 3 次；仍失败进入人工复核和对账 |
| 前端状态 | 页面展示结果 | 定义加载态、空态、错误态、无权限态、部分成功态和重试入口 |
| 观测 | 记录日志 | 定义业务 key 日志、指标、trace、告警阈值和 dashboard 字段 |
| 验收 | 测试通过 | 定义功能、异常、性能、安全、数据和上线验收标准 |
| 代码结构 | 后端负责订单逻辑 | 明确 OrderService 协调 OrderRepository 与 PaymentClient；禁止 Service 直接访问其他聚合的数据库表 |
| 设计模式 | 使用策略模式处理不同支付方式 | 定义 PaymentStrategy 接口及 AlipayStrategy/WechatStrategy 实现；新增支付方式只需注册新实现类 |
| 伪代码 | 实现幂等控制 | 提供幂等键生成、存储校验（如 Redis SETNX）、业务完成后清理的伪代码或工具类模板 |
| 时序 | 模块 A 调用模块 B 完成操作 | 使用时序图标明同步调用 / 异步消息 / Saga 事务；明确补偿触发条件与回滚代码路径 |
| 测试策略 | 需要写单元测试 | 明确 OrderService.create 需覆盖：正常下单、库存不足、重复请求、并发扣减；使用 Mockito 模拟 InventoryClient |
| 错误处理 | 出错时记录日志 | 定义统一错误处理切面：Controller 层负责异常映射，Service 层抛领域异常，Infrastructure 层记录外部调用日志与 TraceID |
| 工程落地 | 研发自行拆分 | 明确后端任务：T1 领域模型+Repository（order/domain, order/infrastructure），T2 ApplicationService，T3 API+校验，T4 集成测试；标明依赖顺序与代码边界 |

## CV/算法适用条件

系统包含计算机视觉、YOLO、目标检测、模型训练、微调、推理服务、模型服务、GPU/NPU/加速卡选型或算法验收时，应用本节。

## CV/算法评审分类

| 算法分类 | Review items |
|---|---|
| 业务目标与类别定义 | 检测目标、类别列表、正负样例、目标场景、非目标场景 |
| 数据与标注 | 数据来源、数据量、场景分布、训练/验证/测试划分、标注规则、抽检规则 |
| 模型选择与训练计划 | YOLO/模型版本、baseline 对比、预训练权重、超参数、数据增强、可复现性 |
| 指标与验收 | mAP、按类别 precision/recall、漏检率、误报率、延迟、FPS、阈值策略 |
| 硬件与部署 | GPU/NPU/加速卡选择、视频分辨率、路数、batch、显存、FP16/INT8、TensorRT/ONNX、边缘/云端部署 |
| 监控、迭代与风险控制 | 漂移监控、置信度分布、告警量、人工复核、样本反馈、模型版本、回滚 |

## CV/算法一票否决项

以下任一情况触发否决：

- 检测类别或算法业务目标未定义。
- 数据来源、数据量或标注规范缺失。
- 测试集隔离策略缺失。
- 只提供 mAP，没有业务验收指标。
- FPS、推理延迟或并发路数目标缺失。
- 硬件/加速卡选型缺少吞吐、延迟、成本或容量依据。
- 模型版本、训练配置或部署格式缺失。
- 未处理误报、漏报、漂移或模型退化。
- 当业务决策依赖模型输出时，缺少人工复核或样本反馈闭环。

## CV/算法坏味道/好味道示例

| 领域 | 坏味道 | 好味道 |
|---|---|---|
| 目标 | 用 YOLO 检测目标 | 检测 helmet、no_helmet、vest、person_intrusion；小目标召回 >= 90% |
| 类别 | 检测人、车、异常物 | 定义 person、helmet、no_helmet、vest、truck，并给正负样例 |
| 数据 | 后续采集一些图片 | 8 万历史图片 + 2 万新图片，覆盖 12 个摄像点 |
| 划分 | 随机 8:1:1 | 按摄像点划分；测试集包含未见过点位 |
| 模型 | 使用 YOLO | Ultralytics YOLOv8.2、yolov8s.pt、input 640，并与 RT-DETR 对比 |
| 训练 | 微调预训练模型 | 冻结 backbone 20 epoch，全量训练 80 epoch，batch 32，cosine learning rate |
| 指标 | mAP >= 0.8 | no_helmet recall >= 90%，误报率 <= 5%，P95 latency <= 80ms |
| 阈值 | confidence = 0.5 | 阈值来自 PR 曲线；person 0.45，helmet 0.6 |
| 后处理 | 检测到就告警 | 连续 5 帧命中且目标面积达阈值才告警 |
| 硬件 | 使用 GPU 服务器 | 24 路 1080p@25FPS；RTX 4090 支持 12 路，P95 65ms，利用率 70% |
| 部署 | 导出模型部署 | PyTorch 训练，ONNX 导出，边缘节点 TensorRT FP16 engine |
| 迭代 | 后续再训练 | 每月 FP/FN 样本复盘，数据集版本化，灰度发布，可回滚 |

## 问题严重度

| 严重度 | 使用场景 | 典型影响 |
|---|---|---|
| Critical | 否决项、安全/数据重大风险、实现无法开始 | 最终结论为不通过 |
| High | 阻塞核心实现路径，但可在评审周期内修复 | 阻塞开工准备度 |
| Medium | 可能导致返工或测试歧义 | 有条件整改 |
| Low | 提高清晰度，不阻塞实现 | 建议优化 |

## 证据标准

每个低于 3 分的分类必须提供：

- 缺失或薄弱的文档证据。
- 为什么重要。
- 对实现、测试或上线的影响。
- 必需或建议整改。
- 能关闭问题的验收证据。
