# 软件研发 FMEA 风险评审

## 概览

面向软件研发场景执行 FMEA 风险评审，覆盖需求、设计、开发、测试、发布和运维闭环。核心原则：风险结论必须可追溯、可评分、可整改、可验证；不得凭空制造风险，也不得因需求描述正常而跳过异常链路分析。

详细评分锚点和风险模式库见 `references/software-fmea-reference.md`。当需要评分、判定 AP、识别典型软件失效模式时，先读取该参考文件。

增量知识库补充入口见 `references/knowledge-base-extension.md`。当用户提供企业历史缺陷、线上事故、行业合规要求、平台架构约束、专项风险模式或希望扩展本技能知识库时，读取并维护该文件。

正式输出报告前读取 `references/report-template.md`，并按模板保存完整 Markdown 报告。

## 适用边界

使用本技能处理：

- PRD、用户故事、业务流程、验收标准、需求变更。
- 架构设计、接口协议、数据模型、状态机、权限模型、异步任务、第三方依赖。
- 上线方案、灰度策略、回滚方案、数据迁移、配置变更。
- 事故复盘中的潜在失效链补充与预防措施制定。

不使用本技能处理：

- 没有明确功能、流程或工程约束的纯创意讨论。
- 只需要代码实现、普通 bug 修复或单元测试补充的任务。
- 没有可分析对象且用户拒绝补充背景的信息。

## 输入检查

开始分析前，确认是否具备以下信息。缺失时不要臆测，先输出“澄清清单”。

| 信息 | 最低要求 |
|---|---|
| 分析对象 | 需求、设计、接口、发布方案或事故场景 |
| 边界 | 包含范围、排除范围、调用方、依赖系统 |
| 预期功能 | 正常输入、输出、状态变化、性能或合规约束 |
| 业务影响 | 用户、资金、数据、合规、可用性影响 |
| 现有控制 | 评审、测试、监控、告警、灰度、回滚、权限校验 |

如果输入很长，按模块分段评审，并在每段报告中记录“本段范围”和“未覆盖范围”。

## 工作流

严格按以下顺序执行。可以压缩文字，但不能跳过判断。

1. **策划与边界**
   明确分析对象、业务目标、用户群体、系统边界、外部依赖、约束条件和评审假设。

2. **结构分析**
   拆解为系统、模块、功能点、接口、数据对象、状态流、作业任务和运维动作。标出上下游依赖和关键路径。

3. **功能分析**
   为每个最小分析单元定义预期功能、成功条件、验收标准、异常路径、边界条件和现有控制手段。

4. **失效分析**
   对每个关键分析单元推导 FE/FM/FC：
   - FE：失效影响，分别说明用户、业务、系统、数据、合规影响。
   - FM：失效模式，例如功能丧失、错误执行、越权、重复执行、状态错乱、性能退化、数据不一致。
   - FC：失效原因，例如规则缺失、边界未定义、状态机不完整、并发竞争、依赖失败、监控缺失。

5. **风险分析**
   使用 S/O/D 评分锚点给出 1-10 分，并说明评分依据。AP 使用 S/O/D 联合判定，不只使用 RPN。若无法可靠评分，标记“需补充信息”，不要伪造精确分数。

6. **优化措施**
   每个中高风险至少给出预防措施和探测措施：
   - 预防措施：修改需求、设计约束、接口契约、状态机、权限校验、幂等、事务、限流、降级、回滚。
   - 探测措施：测试用例、自动化校验、日志、指标、Trace、告警、审计、灰度观测。

7. **闭环验证**
   为每项措施定义负责人角色、验证方式、验收证据和状态。没有验证方式的整改建议不算闭环。

8. **结果保存**
   将完整FMEA评审结果保存为Markdown文件。不能只在对话中输出摘要、Top风险或整改建议。

9. **知识库沉淀**
   当评审中发现可复用的软件风险模式、评分修正依据、历史事故教训、行业约束或企业级控制措施时，询问或根据用户要求将其追加到 `references/knowledge-base-extension.md`。沉淀内容必须结构化、可追溯，不能把一次性项目细节直接写成通用规则。

## AP 判定原则

使用 `H/M/L`：

- `H`：需立即整改或阻断进入下一阶段。
- `M`：需在当前阶段补充控制措施并复核。
- `L`：可接受但需记录依据和必要监控。

判定规则：

- S >= 9，通常为 H；除非有强预防控制和强探测证据。
- S >= 7 且 O >= 4，通常为 H 或 M，取决于 D。
- S >= 7 且 D >= 7，至少为 M。
- O >= 7 且 D >= 6，至少为 M，即使 S 中等。
- 涉及资金、隐私、权限、合规、数据不可逆损坏、核心链路不可用时，不得仅因发生概率低而降为 L。
- 禁止只按 RPN 排序；RPN 可作为辅助信息，但 AP 必须说明 S/O/D 组合理由。

## 知识库补充入口

使用 `references/knowledge-base-extension.md` 作为可维护的增量知识库。适用内容包括：

- 企业历史线上事故、缺陷复盘和典型逃逸缺陷。
- 行业/业务域风险模式，例如支付、交易、SaaS、多租户、IoT、AI Agent、数据平台、医疗或工业控制软件。
- 公司内部评分修正规则、发布门禁、合规要求、安全基线和SRE运行约束。
- 可复用的预防控制、探测控制、测试策略、监控指标和验收证据样例。

补充规则：

- 新知识必须写明适用场景、触发信号、典型FE/FM/FC、建议S/O/D倾向、推荐控制措施和来源/证据。
- 与 `software-fmea-reference.md` 冲突时，不直接覆盖基础规则；在增量知识库中记录“覆盖范围”和“优先级原因”。
- 不写入敏感凭据、客户隐私、真实个人信息或无法脱敏的生产数据。
- 对一次性项目结论，只作为案例记录；只有可复用时才抽象为通用风险模式。

## 执行约束

- 不得用“正常情况下”“理论上不会”回避异常、并发、依赖失败和边界场景。
- 不得输出没有证据的确定性结论；证据不足时输出澄清项或假设。
- 不得强行制造风险；若某项检查未发现显著风险，记录“已检查，未发现显著风险”及依据。
- 对资金、权限、隐私、合规、数据一致性、核心可用性场景必须分析极端失效。
- 每项中高风险必须具备可执行整改和可验证验收证据。
- 最终报告必须提示：AI 评审结果为专业参考，仍需产品、研发、测试、安全、运维或法务按场景复核。
- 正式FMEA结果必须落盘保存；最终回复中必须包含报告文件路径。
- 用户要求补充知识库或评审产生可复用经验时，必须使用 `references/knowledge-base-extension.md` 作为入口，不要散落到临时报告中。

## 常见错误

| 错误 | 修正 |
|---|---|
| 只列风险标题，不写 FE/FM/FC | 补全影响、模式、原因，形成因果链 |
| 评分无依据 | 引用需求证据、历史缺陷、控制措施或明确假设 |
| 把测试建议当作预防措施 | 区分预防和探测；测试只能发现，不能消除原因 |
| 只看功能逻辑 | 增加 NFR、安全、数据、可恢复性、可观测性检查 |
| 只有整改建议，没有验证 | 补充 Owner、验证方式、验收证据和状态 |

## G100 子代理集成契约

当本 Skill 由 `g100-requirements-entry` 编排层作为子代理调用时，遵循以下契约。

### 输出路径

固定输出到 requirements 阶段评审目录：

```
artifacts/reviews/001b-fmea-risk-review.md
```

### 执行深度参数

子代理启动时，G100 编排层根据 `final_mode` 传入执行深度：

| final_mode | 执行深度 | 覆盖范围 | 步骤要求 |
|---|---|---|---|
| `fast` | 不启动本 Skill | — | G103 内建风险分析替代 |
| `standard` | `lightweight` | top 3-5 高风险功能 | 12 种失效模式 checklist 扫描 + S/O/D 评分 + AP 判定 + 整改清单（预防/探测/Owner/验证） |
| `complete` | `full` | 全部 in-scope MVP 功能 | 全 9 步工作流 + NFR 专项 + 风险热力图 + 知识库沉淀 |

### 输入文档包

G100 传入的 `stage_documents` 包含：

1. `artifacts/requirements/001-requirements-intake.md`
2. `artifacts/requirements/003-requirements-baseline.md`
3. `artifacts/requirements/004-mvp-definition.md`
4. `artifacts/reviews/001a-prd-design-readiness-review.md`（PRD 准入评审报告，含 failed_blockers 和 required_prd_actions）

### evidence 产出要求

当由 G100 编排层调用时，`evidence/RA-07b/` 目录非空，至少包含 1 份过程证据文件：
- 12 种失效模式 checklist 扫描记录（覆盖了哪几种、哪些功能）
- S/O/D 评分明细和依据（每项风险的评分理由）
- AP 判定过程（H/M/L 的判定逻辑）

### 供 G100 消费的输出字段

评审完成后，G100 编排层从报告中消费以下结构化字段：

| 字段 | 用途 |
|---|---|
| `fmea_summary.execution_depth` | 记录执行深度（`lightweight` / `full`） |
| `fmea_summary.risk_items_total` | 风险项总数 |
| `fmea_summary.ap_h_count` | AP=H 数量（用于门禁判定） |
| `fmea_summary.ap_m_count` | AP=M 数量 |
| `fmea_summary.top_risks` | Top 3 高风险摘要 |
| `fmea_summary.nfr_conclusion` | NFR 专项检查结论 |
| `fmea_summary.remediation_open_count` | 未闭环整改项数量 |
| `ap_h_items[].prevention` | 每个 AP=H 项的预防措施（门禁硬性要求） |
| `ap_h_items[].detection` | 每个 AP=H 项的探测措施（门禁硬性要求） |

### 门禁约束

1. `lightweight` 模式应控制在 `45-60` 分钟内完成。
2. `full` 模式应控制在 `90-150` 分钟内完成。
3. 超时需在报告中记录 `resume_from` 和未覆盖范围。
4. AP=H 风险项缺少预防措施或探测措施时，G100 编排层将拒绝推进到 `RA-07c`（GS-Review）。
