# 软件研发 FMEA 参考库

## S/O/D 评分锚点

### S 严重度

| 分值 | 软件研发判定锚点 |
|---:|---|
| 10 | 违法合规、重大资金损失、大规模隐私泄露、核心数据不可逆损坏、系统级长时间不可用 |
| 9 | 核心业务不可用、订单/支付/认证/授权主链路失败、关键数据丢失 |
| 8 | 主流程严重受损，用户无法完成关键任务，需人工批量修复 |
| 7 | 大范围功能异常、状态错乱、明显数据不一致，影响多个团队或大量用户 |
| 6 | 功能部分失败，存在可恢复业务损失或明显运营成本 |
| 5 | 用户体验明显下降，性能、延迟、错误提示影响使用但不阻断主流程 |
| 4 | 局部可恢复异常，重试、刷新或人工小范围处理可恢复 |
| 3 | 轻微功能或展示异常，不影响关键流程和数据正确性 |
| 2 | 极低影响，非关键提示、非核心统计或内部展示问题 |
| 1 | 无用户感知，无业务影响，仅日志或内部诊断级别 |

### O 频度

| 分值 | 软件研发判定锚点 |
|---:|---|
| 10 | 必然发生；需求规则直接导致失效或所有用户路径都会触发 |
| 9 | 极高概率；高频主链路、无控制措施、已有同类事故 |
| 8 | 高概率；复杂状态/并发/依赖场景常见，控制不足 |
| 7 | 较高概率；多系统依赖、边界多、历史缺陷率高 |
| 6 | 中等偏高；部分用户或特定数据条件经常触发 |
| 5 | 中等；有一定控制，但异常路径覆盖不足 |
| 4 | 中低；依赖少，触发条件明确，已有部分防护 |
| 3 | 较低；边缘场景，已有较完整需求和测试控制 |
| 2 | 很低；罕见组合条件且有强约束 |
| 1 | 极低；设计上基本消除原因或有形式化强校验 |

### D 探测度

| 分值 | 软件研发判定锚点 |
|---:|---|
| 10 | 发布前几乎无法发现，线上发生后也难以及时感知 |
| 9 | 主要依赖用户投诉、人工排查或事故后复盘发现 |
| 8 | 线上监控弱，仅通过间接指标或少量日志发现 |
| 7 | 需要专项人工测试或复杂造数，常规测试难覆盖 |
| 6 | 可通过集成测试/压测/安全测试发现，但当前未明确 |
| 5 | 可通过常规测试设计发现，但需求验收标准不完整 |
| 4 | 自动化测试、契约测试或灰度指标较容易发现 |
| 3 | 需求评审或静态检查即可发现，多数团队能识别 |
| 2 | 有强校验、强类型、约束、告警或发布门禁 |
| 1 | 设计上直接阻断失效，且有自动验证和实时告警 |

## 常见软件失效模式库

| 风险模式 | 典型表现 | 常见影响 |
|---|---|---|
| 边界未定义 | 空值、长度、金额精度、时区、分页、超时、重试次数缺失 | 数据异常、崩溃、错误计费 |
| 状态机不完整 | 非法跳转、逆向流程缺失、终态可修改、并发状态冲突 | 订单卡死、库存错乱、审批绕过 |
| 权限模型缺失 | 角色边界模糊、租户隔离不足、对象级授权缺失 | 越权访问、数据泄露、合规事故 |
| 幂等缺失 | 重复提交、重试重复扣款、消息重复消费 | 资金损失、重复订单、数据污染 |
| 事务边界错误 | 本地事务与远程调用混用、补偿缺失、部分成功 | 数据不一致、人工修复 |
| 依赖失败未处理 | RPC 超时、第三方限流、缓存穿透、队列堆积 | 主链路不可用、雪崩 |
| 并发竞争 | 库存扣减、余额更新、配置热更新、任务抢占 | 超卖、重复执行、脏写 |
| 数据迁移风险 | 字段兼容、默认值、回填、双写校验缺失 | 历史数据损坏、查询错误 |
| 可观测性不足 | 无关键日志、Trace 断链、指标缺失、告警阈值缺失 | 故障发现晚、定位慢 |
| 回滚不可用 | 数据结构不可逆、配置无版本、灰度无隔离 | 事故扩大、恢复时间长 |
| 安全输入风险 | 注入、XSS、文件上传、反序列化、开放跳转 | 安全漏洞、数据泄露 |
| 合规遗漏 | 数据保留、脱敏、用户授权、审计记录缺失 | 法务风险、监管处罚 |

## 角色视角检查

| 角色 | 重点问题 |
|---|---|
| 产品 | 规则是否完整，用户路径是否闭合，异常提示是否明确 |
| 后端 | 状态、事务、幂等、并发、依赖失败、数据一致性 |
| 前端 | 输入校验、交互状态、重复提交、兼容性、错误恢复 |
| 测试 | 边界、反向路径、组合场景、自动化覆盖、验收标准 |
| 安全 | 权限、认证、授权、租户隔离、敏感数据、注入风险 |
| 运维/SRE | 监控、告警、容量、灰度、回滚、故障演练 |
| 数据 | 口径、血缘、迁移、回填、质量校验、审计 |
| 法务/合规 | 隐私授权、数据保留、跨境、审计、监管要求 |
