# SSOT 审计与索引建立

评估项目中的唯一事实源体系，生成按问题类型组织的 SSOT-MAP.md 速查索引，并挂载到项目入口文件。

## SSOT 准入标准（6 条）

一项文件/目录要成为 SSOT，需通过以下判定。用这 6 条逐项评估每个候选源：

| # | 标准 | 判定方法 | 通过表现 |
|---|------|---------|---------|
| A1 | **权威性** | 该文件是某类信息的"定义者"还是"派生者"？ | 下游消费方都从它派生，不反向修改 |
| A2 | **无冗余** | 同一信息在 repo 中是否只存一份？ | 其余位置只有 link/pointer，没有 copy |
| A3 | **可追溯** | 下游制品能否反向定位到上游 SSOT？ | 修改 SSOT 时能识别所有受影响方 |
| A4 | **可验证** | SSOT 内容能否通过工具机械验证正确性？ | 编译、lint、diff、契约一致性检查可通过 |
| A5 | **版本受控** | 是否纳入 git 且变更历史可审计？ | `git log` 可见完整历史 |
| A6 | **能落地** | 内容是否足够精确，能直接驱动下游动作？ | 可直接用于代码生成、建表、接口联调 |

**判定流程**：对每个候选源，用 6 条逐一打分（通过/不通过/部分通过）。全部通过或仅 A4 部分通过 → 金级 SSOT。A1/A2/A6 通过但 A4 弱 → 银级 SSOT。仅 A1 通过其余弱 → 铜级 SSOT（运行时/过程态）。A1 不通过 → 非 SSOT（派生制品）。

## SSOT 分级规则

| 等级 | 特征 | 处理方式 |
|------|------|---------|
| **金级 SSOT** | 权威且唯一，可直接驱动编码。A1~A6 全部通过（A4 可为部分通过） | 写入 SSOT-MAP.md 作为主入口；是代码生成的输入源 |
| **银级 SSOT** | 设计权威，驱动下游阶段。A1/A2/A6 通过，A4 弱（需人工审查） | 写入 SSOT-MAP.md 作为设计依据；标注"人工审查" |
| **铜级 SSOT** | 运行时/过程态，持续更新。A1 通过但无机械验证 | 写入 SSOT-MAP.md 作为追踪入口；标注"过程态" |
| **非 SSOT** | 派生制品，从上游 SSOT 生成或拷贝而来。A1 不通过 | **不写入** SSOT-MAP.md；在问题清单中标注为冗余 |

**区分派生制品 vs SSOT 的关键判断**：如果 B 文件的全部内容都可以从 A 文件自动生成或推导，则 B 是派生制品，A 是 SSOT。如果两份文件内容相同但存储在两个位置，只保留一份为 SSOT，另一份应替换为指针文件。

## 问题检测规则（5 类）

审计过程中必须主动检测以下问题：

| 类型 | 检测方法 | 严重度 |
|------|---------|--------|
| **双副本** | 同一内容在两个目录各有一份，校验和相同但无指针关系 | 中 — 未来必分裂 |
| **过时引用** | 入口文件中引用的路径在文件系统中不存在 | 低 — LLM 导航失败 |
| **空目录** | 目录存在但仅有 `.gitkeep` 或 README | 低 — 占位，但应标注 |
| **未验证制品** | 内容标注为"草稿"或无法通过机械验证（编译、lint） | 中 — 下游可能遇到错误 |
| **无索引** | SSOT 目录缺少 `_index.md` 或覆盖清单，LLM 无法快速定位目标文件 | 低 — 降低 LLM 工作效率 |

## 执行流程

### Step 1: 探索项目结构

1. 读取项目入口文件（CLAUDE.md / AGENTS.md / GEMINI.md / README.md），提取已声明的 SSOT 位置
2. 用 Glob 遍历声明的 SSOT 目录，列出完整文件树
3. 用 Glob 遍历 `contracts/`、`docs/` 等关联目录
4. 读取各目录下的 `_index.md` 和关键交接文档（handoff summary）

### Step 2: 逐目录评估

对每个候选源目录，执行评估：
1. 用 6 条准入标准逐项打分
2. 确定等级（金/银/铜/非SSOT）
3. 对派生制品追溯其上游 SSOT
4. 记录评估结果

### Step 3: 检测问题

运行 5 类问题检测：
1. **双副本**：对比关键文件的内容一致性（`diff` 或校验和）。如果 `contracts/` 和 `artifacts/` 下有内容完全相同的文件 → 标记为双副本
2. **过时引用**：验证入口文件中所有路径引用的目录/文件是否存在
3. **空目录**：检查仅有占位文件的目录
4. **未验证制品**：识别标注为"草稿"或缺乏编译/验证记录的文件
5. **无索引**：检查每个 SSOT 目录是否有 `_index.md` 或覆盖清单

### Step 4: 生成 SSOT-MAP.md

输出文件位置：用 `AskUserQuestion` 询问用户，提供默认值 `docs/SSOT-MAP.md`。

文件结构（按此模板）：

```markdown
# SSOT 速查索引

LLM 按需查找权威设计制品的导航入口。**本文件只存路径指针，不存内容——内容权威性仍在原文件。**

## 按问题类型查找

### 我要理解业务术语 →
- `<path>` — <一句话说明>

### 我要知道系统要做什么（需求） →
- `<path>` — <一句话说明>

### 我要知道系统怎么划分（架构） →
- `<path>` — <一句话说明>

### 我要知道为什么这样设计（决策依据） →
- `<path>` — <一句话说明>

### 我要知道具体用什么技术栈 →
- `<path>` — <一句话说明>

### 我要写某个组件的代码 →
- `<path-to-index>` — <组件总览>
- `<path-to-component-pattern>` — <具体组件路径模式>

### 我要实现某个接口 →
- `<path-to-index>` — <接口总览>

### 我要建数据库表/定义数据结构 →
- `<path-to-index>` — <数据对象总览>

### 我要知道阶段间如何衔接 →
- `<path>` — <交接文档>

### 我要知道设计评审结论和质量门结果 →
- `<path>` — <评审报告>

### 我要追溯需求到设计到实现的全链路 →
- `<path>` — <追溯矩阵>

## 按制品编号快速定位

| 编号前缀 | 含义 | 定位路径 |
|---------|------|---------|
| ... | ... | ... |

## 维护规则

1. 新增制品类型时更新
2. 目录结构变更时更新
3. 编号体系扩展时更新
4. 本文件不超过 100 行
```

**约束**：
- 只写入金级和银级 SSOT 的路径指针（铜级按情况决定，非 SSOT 不写入）
- 每个路径附带一句话说明，帮助 LLM 判断是否需要进一步读取
- 制品编号速查表只列出有明确编号前缀的制品，无编号体系则省略
- 总行数控制在 100 行以内

### Step 5: 更新项目入口文件

自动更新 CLAUDE.md（或等效入口文件）：

1. **导航树**：在文档导航区添加 `SSOT-MAP.md` 条目，说明其定位
2. **维护规则**：新增或前置一条规则，描述 SSOT-MAP.md 的作用和更新时机
3. **过时引用修复**：将在 Step 3 中检测到的过时引用替换为实际存在的路径

新增的维护规则模板：

```markdown
0.`docs/SSOT-MAP.md` 是 LLM 按问题类型查找权威设计制品的速查索引，只存路径指针不存内容。更新时机：① 目录结构变更时 ② 新增制品类型时 ③ 编号体系扩展时 ④ 发现索引遗漏或过时路径时。每次更新后检查不超过 100 行
```

### Step 6: 输出审计摘要

向用户报告：
1. SSOT 分级结果（金/银/铜各多少个）
2. 发现的问题及严重度
3. SSOT-MAP.md 的生成位置
4. CLAUDE.md 的变更摘要
5. 建议的后续动作（如清理双副本、补全空目录）

## 注意事项

- **索引不是 SSOT**：SSOT-MAP.md 只有路径指针，不是内容的权威源
- **只做检测，不主动清理**：发现双副本时报告并建议，不要直接删除文件。用户的明确确认后再执行清理
- **尊重现有命名**：制品编号体系（CMP-XXX、IFC-XXX 等）沿用项目的现有编号，不要发明新编号
- **入口文件优先**：如果项目有 CLAUDE.md / AGENTS.md / GEMINI.md，以 CLAUDE.md 为第一优先级。不要创建新的入口文件
